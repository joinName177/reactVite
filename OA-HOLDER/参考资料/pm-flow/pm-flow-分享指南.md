# Cursor /pm-flow 使用与分享指南

---

## 核心结论（金字塔顶端）

**`/pm-flow` 让 Cursor 按产品经理完整流程处理需求，避免「先写代码再返工」—— 一句话需求即可触发，产出从澄清到验收的完整链路。**

```
一句话需求 → P0 澄清 → 01/02/03 文档 → 可交互原型 → 自动化验收
```

---

# 第一部分：为什么值得用

## 一、价值概览

| 维度 | 核心价值 |
|------|----------|
| **个人** | 减少返工、产出标准化、降低心智负担、技能复用 |
| **团队** | 流程一致、可追溯、质量门槛、知识沉淀 |
| **协作** | 产品↔研发对齐、自动化验收、问题交接 |

## 二、价值详解

### 2.1 对个人

| 价值 | 说明 |
|------|------|
| 减少返工 | 先澄清再动手，避免「写完了发现需求不对」 |
| 产出标准化 | 01/02/03 文档结构固定，便于沉淀和复用 |
| 降低心智负担 | 不用每次重复说明「你要先问清楚再写」，AI 自动遵守 |
| 技能复用 | 一次配置，多项目、多电脑通用 |

### 2.2 对团队

| 价值 | 说明 |
|------|------|
| 流程一致性 | 所有人用同一套 PM 流程，评审和交接更顺畅 |
| 可追溯 | 需求卡 → 方案 → 功能清单 → 原型 → 验收，链路清晰 |
| 质量门槛 | 功能清单自检、专家审查、03↔原型对照，减少遗漏 |
| 知识沉淀 | 规则和技能即「可执行的 SOP」，新人也能快速上手 |

### 2.3 对协作

| 价值 | 说明 |
|------|------|
| 产品 ↔ 研发 | 03 功能清单 + 验收用例，研发有明确实现依据 |
| 自动化验收 | Playwright 脚本可回归，减少手工点测 |
| 问题交接 | 验收问题表可直接交给研发修复，信息不丢失 |

### 2.4 解决的问题 vs 做法

| 问题 | /pm-flow 的做法 |
|------|-----------------|
| 需求不清就开干 | 先输出 P0 澄清问题清单，等你回答后再进入方案 |
| 方案没确认就写原型 | 必须等你确认方案后才产出功能清单和原型 |
| 原型和文档脱节 | 原型严格按 03-功能清单逐条实现，并做「03 ↔ 原型」对照 |
| 验收靠人工 | 可生成 Playwright 验收脚本，自动化回归 |

# 第二部分：怎么用

## 三、如何触发

在 Cursor 对话中输入以下**任一**方式即可触发：

| 方式 | 示例 |
|------|------|
| 显式命令 | `/pm-flow` 或 `/pm-flow 一句话需求: 添加导出 Excel 功能` |
| 自然语言 | `一句话需求: 添加导出 Excel 功能` |
| 关键词 | `按 PM 流程`、`按默认协议`、`帮我分析需求`、`走需求流程` |

### 一句话模板（可直接复制）

```
一句话需求: [这里写需求描述]
```

带可选补充：

```
一句话需求: [需求描述]
可选补充: 现有页面 @原型/模块/xxx，目标用户 xxx，约束 xxx
```

## 四、交付物说明

| 文档 | 说明 |
|------|------|
| 01-需求卡 | 背景、用户、成功标准、约束 |
| 02-需求方案 | 多方案对比、流程图、推荐方案 |
| 03-功能清单与验收标准 | 原子化功能列表、验收用例 TC、自检表 |
| 可交互原型 | HTML 原型，按 03 逐条实现 |
| 验收脚本 | `acceptance.spec.mjs`（Playwright） |
| 验收问题表 | 失败用例、复现步骤、交给研发修复 |

## 五、如何在新电脑 / 新项目中使用

### 5.1 需要复制的文件

| 类型 | 路径 | 作用 |
|------|------|------|
| 项目级 | `.cursor/commands/pm-flow.md` | 命令定义 |
| 项目级 | `.cursor/rules/product-requirement-flow.mdc` | PM 流程规则 |
| 用户级 | `~/.cursor/skills/product-manager/SKILL.md` | 产品经理技能 |
| 用户级 | `~/.cursor/skills/product-expert/SKILL.md` | 产品专家技能 |

### 5.2 快速安装步骤

1. **复制项目级**：将 `.cursor/commands/` 和 `.cursor/rules/` 复制到新项目
2. **安装用户级**：`cd ~/.cursor/skills && tar -xzvf /path/to/pm-flow-skills.tar.gz`
3. **验证**：在 Cursor 中输入 `/pm-flow`，应看到 P0 澄清问题清单

### 5.3 分享包说明

项目根目录的 `pm-flow-skills.tar.gz` 已包含 product-manager 和 product-expert 技能，可直接分发。

## 六、使用技巧

| 技巧 | 说明 |
|------|------|
| 需求描述具体 | ✅ `在日历视图中支持拖拽事件改期` ⚠️ `做个日历功能` |
| 善用可选补充 | 补充目标用户、约束、现有页面引用 |
| 分阶段确认 | P0 → 02 方案 → 03 功能清单，每步确认后再进入下一步 |
| 可跳过环节 | 自动化验收、问题表交接可延后，AI 会提供替代方案 |

# 第三部分：怎么扩展

## 七、Flow、Rule、Skill 的区别

| 维度 | **Flow（流程）** | **Rule（规则）** | **Skill（技能）** |
|------|------------------|------------------|-------------------|
| 是什么 | 业务上的「做什么、按什么顺序做」 | Cursor 里的「何时生效、必须/禁止做什么」 | Cursor 里的「怎么做、用什么方法」 |
| 存在形式 | 概念/设计，体现在 Rule + Skill 里 | `.cursor/rules/*.mdc` | `~/.cursor/skills/*/SKILL.md` |
| 关注点 | 流程设计 | 行为约束 | 领域能力 |

**关系**：Flow 是「要什么流程」，Rule 是「管住 AI」，Skill 是「教会 AI」。

```
Flow（流程） → 被拆成 → Rule（规则）+ Skill（技能）
```

**在 pm-flow 中的分工**：

| Flow 环节 | Rule 负责 | Skill 负责 |
|-----------|----------|------------|
| 何时进入 | ✅ 触发条件 | |
| 第一轮出 P0 | ✅ 禁止直接给方案 | |
| 如何做 P0/01/02/03 | ✅ 产物命名 | ✅ 步骤、模板、自检 |
| 原型按 03 实现 | ✅ 唯一依据 | ✅ 逐条对照 |
| 验收脚本与问题表 | ✅ 要求生成 | |

## 八、Cursor 流程的三要素

| 要素 | 位置 | 作用 |
|------|------|------|
| Commands | `.cursor/commands/*.md` | 用户输入 `/xxx` 时触发的动作说明 |
| Rules | `.cursor/rules/*.mdc` | AI 必须遵守的流程、约束、禁止行为 |
| Skills | `~/.cursor/skills/*/SKILL.md` | 领域知识、步骤、检查清单、正反例 |

配合关系：**Command 触发 → Rule 规定流程 → Skill 提供执行细节**。

## 九、创建新流程的步骤

1. 明确流程目标：要解决什么问题？产出什么？谁确认？
2. 写 Command：`.cursor/commands/xxx.md`，写清触发方式、执行要求
3. 写 Rule：`.cursor/rules/xxx.mdc`，写清何时触发、必须动作、禁止行为
4. 写 Skill（可选）：`~/.cursor/skills/xxx/SKILL.md`，写步骤、模板、自检表
5. 试跑迭代：用真实需求试跑，根据 AI 表现调整

## 十、维护建议

| 维护项 | 建议 |
|--------|------|
| 版本记录 | Rule/Skill 顶部加 lastUpdated，重大变更写 changelog |
| 路径适配 | 规则中的路径可抽成说明，方便不同项目复用 |
| 分享包 | 定期打包 commands + rules + skills，便于团队同步 |
| 反馈收集 | 记录「AI 没按预期执行」的案例，反推 Rule 表述是否模糊 |

## 十一、可复用的流程类型

- **代码审查流程**：输入 PR/文件 → 按检查清单审查 → 输出问题表
- **Bug 分析流程**：输入现象 → 复现步骤 → 根因假设 → 修复建议
- **文档写作流程**：输入主题 → 大纲确认 → 分节撰写 → 自检
- **设计评审流程**：输入设计稿 → 可用性检查 → 改进建议

**核心思路**：把「人脑里的 SOP」写成 Rule + Skill，让 AI 稳定执行。

# 附录：速查与 FAQ

## 文件结构速查

```
项目/
├── .cursor/
│   ├── commands/pm-flow.md
│   └── rules/product-requirement-flow.mdc
~/.cursor/skills/
├── product-manager/SKILL.md
└── product-expert/SKILL.md
```

## 常见问题

| 问题 | 答案 |
|------|------|
| 没有 Cursor 能用吗？ | 不能，依赖 Cursor 的 commands、rules、skills |
| 必须用 Playwright 做验收吗？ | 推荐但可跳过，AI 会提供手动验收说明 |
| 项目没有 `原型/模块/` 怎么办？ | AI 会按规则创建，或修改 Rule 中的路径 |
| 可以只做需求分析不做原型吗？ | 可以，02 确认后说「本期不做原型，只产出 03」即可 |
