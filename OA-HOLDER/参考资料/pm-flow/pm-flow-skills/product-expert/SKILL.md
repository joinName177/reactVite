# 产品专家评审技能

## 输出原则（必须遵守）

- **只写问题，不写通过的项**。通过的检查项一律不输出。
- **每条 ≤30 字**，直指问题与修正建议。
- **总输出 ≤400 字**。问题多时只列最关键的 5～7 条。
- 无问题则输出：「评审通过，无关键问题。」

---

## 评审维度与关键问题

按以下维度检查，**仅输出有问题的项**：

### 流程合规
- Step 4 反证未在 Step 5 前完成 → 指出
- 反证结论为「不必要/可延后」却进入 Step 5～7 → 指出
- 大需求未走完整 7 步 / 小需求未走精简路径 → 指出
- 未做反思 pass → 指出

### 产出完整性
- Step 1：缺价值主张/业务目标/关键指标
- Step 2：缺成功标准或不可验收
- Step 3：用户对象未区分主/次，或「所有人都是目标用户」
- Step 4：缺反证结论，或未考虑错误后路（撤销/删除）
- Step 5：未先列约束，或未考虑 IA 深度（过载、空状态）
- Step 6：缺方案对比表，或某方案缺流程图
- Step 7：缺假设与待澄清、缺「本期不做」

### 逻辑一致性
- 方案无法支撑成功标准
- 主角色/主场景在方案中无对应
- 流程图主路径 >7 步、节点含糊

### 领域适配（多维表）
- 数据建模/权限层级/多租户与现有设计冲突
- 视图、数据源、AI 能力与成功标准不符

### 数据分析专家视角
- 关键指标缺口径定义（统计周期、计算方式、单位）
- 数据质量要求未明确（完整性、准确性、一致性）
- 数据清洗/预处理流程缺失或不可追溯
- 分析逻辑不合理（如用平均值替代中位数、忽略异常值）
- 数据来源与更新频率未说明
- 指标计算方式与业务逻辑不符（如环比/同比混淆）

### BI 专家视角
- 报表/仪表盘结构不合理（总览→分维度→明细）
- 图表类型选择不当（如用饼图展示趋势）
- 缺少多维度下钻能力（时间/区域/产品等）
- 实时性要求与数据更新策略不匹配
- 报表性能优化未考虑（数据量级、查询复杂度）
- 可视化元素过多导致信息过载
- 缺少数据联动与筛选联动设计

### 多维表系统专家视角
- 字段类型选择不当（如用文本存日期、用数字存ID）
- 表间关联关系设计不合理（一对多/多对多）
- 视图配置与业务场景不匹配（表格/看板/日历）
- 权限设计违反多租户隔离原则
- 数据源接入方式与同步策略不明确
- AI 能力应用场景不清晰（字段生成/数据分析/自动化）
- 组织-空间-项目层级权限冲突
- 数据导入/导出格式与兼容性未考虑

### 风险
- 技术/业务依赖未明确，缺降级策略
- 假设与待澄清模糊

### 功能清单专项审查（输入 02 + 03 时执行）

当输入包含 02-需求方案 与 03-功能清单与验收标准 时，按以下维度检查，**仅输出有问题的项**：

| 检查项 | 说明 |
|--------|------|
| 02→03 映射 | 02 各节（含流程图节点）是否已映射到 03 功能条或显式注明「不产出」 |
| CRUD 与主对象 | 主对象的增删改查是否齐全，不做的是否在「本期不做」中显式注明 |
| **可交互操作与 TC** | 功能清单中编辑、删除、轮换、吊销、启用/禁用等可交互操作是否均在验收标准中有对应 TC，且 TC 步骤包含「执行该操作→断言界面/反馈」；对应 TC 是否在自动化脚本中有实现或显式注明不自动化 |
| 隐性功能 | 加载态、错误处理、空状态、权限校验、撤销/回滚（如适用）是否已列出 |
| 关联需求 | 「关联与依赖」小节是否完整（数据来源/去向、共用能力、权限） |
| 本期不做一致性 | 02 与 03 的「本期不做」是否一致；03 新增的是否合理 |
| 03 结构完整性 | 关联与依赖、功能清单、本期不做、验收标准、自检表是否齐全 |
| TC 与功能清单映射 | P0 功能是否有对应 TC；是否存在孤儿 TC 或孤儿功能 |

**输出格式**（功能清单专项审查）：
```markdown
## 功能清单专项审查结论

- **评审通过**：无关键遗漏。
- **或** 存在问题（仅列有问题的项，每条 ≤30 字）：
  1. [02 §X.X] 未映射到功能清单 → 建议补「xxx」
  2. 主对象「xxx」缺「删」→ 建议补或注明本期不做
  3. 隐性功能「空状态」未单独列出 → 建议补
  ...
```
