# 我的编排 Skill 规则

本规则定义如何创建**多 Skill 编排型 Skill**：**先通过对话摸清所有想法，再一次性产出编排 Skill 文档与配套 README**；产出的编排 Skill 文档需按固定顺序串联多个已有 Skill，并包含下文规定的结构与关键点。创建过程遵循与「我的 Skill 创建规则」一致的「先提问、再产出」节奏。

---

## 何时使用本规则

- 你要**新建一个编排型 Skill**，即：按固定顺序调用多个已有 Skill，每步有明确输入/输出与交接。
- 你希望**先和 AI 把编排想法聊清楚**（编排哪几个 Skill、顺序、每步输入输出、提问与文档确认节奏等），再让 AI 产出 SKILL.md 与 README，而不是一上来就写稿。
- 你希望编排的执行节奏统一、可预期（尤其是提问→答复→判断→文档生成前确认→再进入下一步）。

**不触发**：只使用某个已有编排 Skill、或只创建单个非编排型 Skill 时，不适用本规则。

---

## 执行本规则时的必须行为（AI 必读）

当用户请你**按本规则创建某一编排 Skill** 时，你必须遵守以下行为，不得跳过：

### 第一反应：先提问，不写稿

- **你的第一轮回复必须是「提问」**：根据用户的需求描述，先复述或概括用户想做的编排，然后**向用户提出一批问题**，用于了解清楚用户的**所有编排诉求**。**禁止**在第一轮就输出编排 Skill 的 SKILL.md 或 README 全文（也不得输出完整正文）。
- **持续多轮提问直到你没有问题**：用户每答复一批问题后，你只做两件事——（1）根据答复判断是否还有要问的；（2）若还有，**只输出下一批问题**，继续等用户答复；若没有了，**先询问用户**「我这边没有更多问题了，是否要生成本次的编排 Skill 文档？您确认后我再输出。」**只有**用户明确确认（如「可以」「生成吧」「要」）后，才可进入「产出」。
- **禁止在以下任一情况产出编排 Skill 文档**：
  - 你尚未向用户提出过任何问题；
  - 你心中仍有关键编排诉求未问清（见下方「必须问清的编排诉求」）；
  - 用户未表示「想法已摸清」「可以出稿了」「可以写了」或等价意思；
  - 用户未明确确认「要生成」「可以输出」本次文档。

### 必须问清的编排诉求（提问清单）

在产出文档前，你必须通过提问与用户确认以下方面（可多轮、分批问，不必一次问完，但缺一不可）：

| 类别 | 需问清的内容 |
|------|----------------|
| **编排对象与顺序** | 要编排哪几个 Skill？固定顺序是什么？每个 Skill 的路径或名称？ |
| **谁用、什么场景** | 谁会用这个编排 Skill？在什么场景下触发？（如：拖入/@ 并说一句话） |
| **每步输入与输出** | 第一步的输入从哪来（用户提供什么）？每一步的产出物是什么（文档名/形态）？上一步产出如何交给下一步？ |
| **提问与文档确认节奏** | 被编排的 Skill 执行时是否要「先提问、等答复、再判断是否继续问」？是否每步都要「生成文档前确认」（问用户是否要生成本步文档、用户确认后才输出）？是否采用「禁止两种跳过」？ |
| **产出物与路径** | 各步产出是否独立、不汇总？编排 Skill 与 README 的存放路径或命名有无要求？ |

若用户首次描述已包含部分信息，你可先复述已清晰的部分，再**只针对尚未清晰的部分提问**；直到上表所列在你心中均已明确，且用户已确认可以出稿，才可产出编排 Skill 文档。

## 创建编排 Skill 的流程（先提问、再产出）

执行本规则时，**必须**按以下三阶段进行；不得在未完成阶段二（摸清想法）前进入阶段三（产出文档）。

### 流程总览

1. **阶段一：仅讨论（ASK）**  
   用户说明：想做一个什么样的编排、串联哪几个 Skill、谁在什么场景用。AI **只进行分析与提问**，**不生成**最终的编排 Skill 文件。**首轮回复必须是提问**（见上文「第一反应：先提问，不写稿」）。

2. **阶段二：反复提问直到摸清**  
   AI 根据当前描述进行分析，并**主动、连续地提问**（可多轮）。每轮可包含：编排对象与顺序的确认、每步输入/输出的约定、是否要「生成文档前确认」与「禁止两种跳过」、与其它流程的边界等。直到**当前会话内**已摸清用户的**所有编排诉求**（用户确认或等价表述），且**你已没有更多问题要问**。

3. **阶段三：产出交付物**  
   **仅在**用户确认「想法已摸清」「可以出稿了」或等价意思、**且你已询问「是否要生成本次文档」并得到用户明确确认**之后，再产出：
   - **编排 Skill 文档**：即最终的 `SKILL.md`（含 YAML frontmatter 与正文，且符合下文「一～七」的结构与关键点）。
   - **配套 README**：见「我的 Skill 创建规则」中 README 必须包含的内容（使用方式、本次创建中的确认与否定、使用案例、结果示例）。

### 阶段一：仅讨论（ASK）

- 默认假定用户处于**只讨论、不直接要文档**的意图，或先说明将通过多轮提问把需求摸清。
- **范围限定**：只讨论「当前要创建的这一个编排 Skill」，包括要编排哪几个 Skill、顺序、每步输入输出、是否采用统一的提问与文档确认节奏。
- AI 先复述或概括用户的想法，再说明将通过多轮提问把编排需求摸清，然后进入阶段二。

### 阶段二：反复提问直到摸清

- **分析**：根据用户当前描述，提炼出：编排目标、被编排的 Skill 及顺序、每步输入/输出、谁用、触发方式、是否要「生成文档前确认」与「禁止两种跳过」等。
- **提问方式**：  
  - 一次可问多个相关问题，但主题清晰（例如：先问编排对象与顺序，再问每步输入输出）。  
  - 可给出 2～3 个可选方案让用户选「要 / 不要 / 再改」（如：是否每步都做「生成文档前确认」）。  
  - 对模糊处（如「看情况再定」）主动追问成可执行规则。
- **结束条件**：  
  - 用户明确表示「就这些」「没了」「可以写了」等；或  
  - 用户对关键问题（编排对象与顺序、每步输入输出、提问与文档确认节奏、产出物形式）都已做出选择或确认。  
- **记录**：在会话中自然记录用户的**确认与否定**（例如：要编排 A→B 两个 Skill、每步都要先生成前确认、不做汇总文档）。这些将用于 README 的「本次创建中的确认与否定」部分。

### 阶段三：产出交付物

**仅在阶段二结束条件满足后执行。**

- **编排 Skill 文档（SKILL.md）**：内容完全基于**当前会话中已确认**的编排对象、顺序、执行流程、提问与文档确认节奏编写；结构需符合下文「一～七」；不引入未在会话中确认的假设。
- **配套 README**：与「我的 Skill 创建规则」一致，须包含使用方式、本次创建中的确认与否定、使用案例、结果示例（或合并为案例与示例）。
- 若用户指定了存放路径或命名，按用户要求；未指定则给出建议路径。

### 会话中的注意事项

- **不提前写稿**：在用户未确认「想法已摸清」前，不输出最终的、可保存的编排 Skill 的 SKILL.md 全文；可输出提纲或片段用于确认。
- **术语一致**：一旦用户选定了某说法（如「生成文档前确认」「禁止两种跳过」），后续提问与产出的文档中保持一致。
- **一次只做一个编排 Skill**：同一会话内只完成「当前这一个」编排 Skill 的创建；若用户想再做一个，视为新会话、新一次「按本规则创建」。

### 简要自检清单（产出前）

- [ ] 用户已确认或等价表示「想法已摸清」或「可以出稿了」。
- [ ] 编排对象与顺序、每步输入输出、提问与文档确认节奏已在会话中确认。
- [ ] 产出的 SKILL.md 与当前会话结论一致，且符合下文「一～七」的结构与关键点。
- [ ] 配套 README 已包含：使用方式、确认与否定、使用案例、结果示例（或合并后的案例与示例）。

## 一、编排 Skill 文档必须包含的结构

新建的编排 Skill 的 SKILL.md 应包含以下小节（顺序建议保持一致）：

| 小节 | 说明 |
|------|------|
| **何时使用** | 谁在什么场景下会用到本编排；触发方式（如拖入/@ 并说自然语言）。 |
| **编排对象与顺序** | 被编排的 Skill 列表及固定顺序（表格：顺序、被编排 Skill 名称、路径）；顺序不可调换。 |
| **执行流程** | 每一步对应一个被编排 Skill，含：输入、执行、提问与澄清、生成文档前确认、输出、交接。 |
| **关联 Skill 的提问与执行节奏** | 统一规定：先提问→等答复→判断是否还要问→若还要问只继续提问（不输出文档、不进入下一步）→直到没有要问的→询问是否生成本次文档→用户确认后才输出→再进入下一步；并明确禁止的两种跳过。 |
| **编排规则** | 严格顺序、提问节奏、禁止跳过、产出物不汇总、输入来源、路径引用等。 |
| **简要自检** | 每步结束后的检查项（是否已输出该步完整产出、是否满足被编排 Skill 要求的全部小节）。 |

---

## 二、编排对象与顺序（必须明确）

1. **列出被编排 Skill**：用表格列出，列包括：顺序、被编排 Skill 名称、路径（相对或绝对，便于执行前读取）。
2. **固定顺序**：明确说明顺序不可调换；第一步的输入来自用户，后续每步的输入来自上一步的产出。
3. **路径引用**：执行各步前，先读取对应 Skill 文件内容，再按其指令执行。

示例结构：

```markdown
## 编排对象与顺序（固定）

本 Skill **固定顺序**调用以下 N 个 Skill，不可调换：

| 顺序 | 被编排 Skill | 路径 |
|------|--------------|------|
| 1 | XXX | `相对或绝对路径/Skill.md` |
| 2 | YYY | `...` |
```

## 三、执行流程（每步必须包含的要素）

对**每一个**被编排的 Skill，在编排文档中为该步写明：

1. **输入**：本步的输入来源（用户在本轮提供的初始输入 / 上一步产出的完整文档）。
2. **执行**：阅读并严格按该 Skill 的指令执行（执行前先读取该 Skill 文件）。
3. **提问与澄清**：若该 Skill 涉及提问，则：
   - 先向用户提出需要确认或补充的问题；
   - **每次用户答复后**，只做一件事：判断是否还有要问的问题；
   - 若还有：**仅输出下一批问题**，不输出该步文档、不进入下一步；重复直到某次用户答复后判断「没有更多问题要问」；
   - 若没有了：进入「生成文档前确认」。
4. **生成文档前确认**：当判断没有更多问题要问时，**必须先**向用户询问是否生成本步文档（例如：「是否要生成本次的『XXX』文档？您确认后我再输出。」）；**只有**用户明确确认（如「可以」「生成吧」「要」）后，才执行「输出」。
5. **输出**：按该 Skill 规定的格式与结构，输出完整的本步产出物（如 Markdown 文档）。
6. **交接**：将本步产出作为下一步的输入；**只有本步完整输出后**才可进入下一步。

**禁止**：

- 在还有要追问的问题时输出当前步文档或进入下一步。
- 在未询问「是否要生成本次文档」且未得到用户明确确认前，输出该步文档。

## 四、关联 Skill 的提问与执行节奏（必须遵守）

编排文档中必须单独成节、明确写出以下节奏，并标注「必须遵守」：

### 1. 多轮提问：每次用户答复后，只做「判断 + 要么继续问、要么请求确认」

- **先提问**：有需要确认或补充的问题时，先在本对话中提出；不输出该 Skill 的最终文档，也不进入下一个 Skill。
- **等用户回答**：用户回答后，根据答复判断「是否还有要问的问题」。
  - **若还有**：仅输出下一批问题，再次等待用户回答；**不得在本轮输出文档，也不得进入下一个关联 Skill**；如此反复，直到某次用户答复后判断「没有更多问题要问」。
  - **若确实没有了**：进入「生成文档前确认」步骤，**不得在本轮直接输出文档**。
- **禁止**：在仍有关键信息想确认或可追问时，不得输出当前 Skill 的文档，也不得询问是否生成文档；必须先把所有要问的问完。

### 2. 生成文档前必须经用户确认

- 当判断**没有更多问题要问**时，必须先向用户询问，例如：「是否要生成本次的 [本步文档名称] 文档？您确认后我再输出。」
- **只有**在用户明确表示「可以」「生成吧」「要」「确认」等同意生成后，才输出该关联 Skill 规定的完整文档。
- 若用户说「还有补充」或提出新信息，则回到多轮提问节奏，继续提问直至再次「没有更多问题」→ 再次询问是否生成文档。

### 3. 当前 Skill 完全结束后再进入下一个

- 只有当前关联 Skill **已输出其完整产出物**（且在用户确认生成之后输出）后，才能进入下一个关联 Skill。
- **禁止**在以下任一情况进入下一步：还有想追问的问题未问完、尚未询问是否生成文档、用户尚未确认要生成本次文档、尚未输出当前 Skill 的文档。

**总结句（建议写入编排文档）**：先问 → 用户答 → 仅判断是否还要问 → 若还要问则只输出下一批问题（不输出文档、不进入下一步）→ 重复直到没有要问的 → 问用户「是否要生成本次文档」→ 用户确认后再输出文档 → 然后才进入下一个关联 Skill。

## 五、编排规则（必须遵守的条目）

在编排 Skill 文档中，以「编排规则」为标题，列出至少以下条目（可根据具体编排增加）：

1. **严格顺序**：必须先完成第 K 步并得到其产出物，再以该产出物为输入执行第 K+1 步；不得跳过任一步或调换顺序。
2. **提问节奏**：每个关联 Skill 的提问均按「关联 Skill 的提问与执行节奏」执行——先提问、等答复、再判断是否继续问（若还要问则只提问不输出）、没有要问的再询问是否生成文档、用户确认后才输出文档；当前 Skill 完全结束后再进入下一个。
3. **禁止两种跳过**：（1）还有问题要问时，不得输出当前 Skill 的文档，也不得进入下一个关联 Skill；必须一直问到没有要问的为止。（2）未先询问「是否要生成本次文档」并得到用户明确确认前，不得输出该次文档。
4. **产出物不汇总**：每一步产出该步规定的文档；按顺序生成即可，不需要再写一份汇总文档（除非你特别约定要汇总）。
5. **输入来源**：第一步的输入来自用户在本轮提供的内容；第 N 步（N≥2）的输入来自第 N-1 步的完整输出文档（由执行方带入上下文）。
6. **路径引用**：执行各步前，先读取对应 Skill 文件内容，再按其指令执行；路径见「编排对象与顺序」表格。

## 六、简要自检（每步结束后）

编排文档中应为**每一步**写一条自检项，例如：

- **第 K 步结束**：已输出完整的「XXX」文档（含该被编排 Skill 要求的全部小节）；且已无待澄清项或用户已确认可进入下一步。

## 七、创建编排 Skill 时的 YAML 与 README

- **YAML frontmatter**：`name`、`description` 必填；description 中建议概括「按固定顺序编排哪几类/哪几个 Skill」「从什么输入到什么产出」「谁在什么场景用」。
- **配套 README**：按「我的 Skill 创建规则」要求，README 须包含：使用方式、本次创建中的确认与否定、使用案例、结果示例（或合并为使用案例与结果示例）。在使用方式中，建议写清与 AI 的配合要点（尤其是：每步多轮提问直到没有要问的、生成文档前必须经用户确认、禁止两种跳过）。

## 执行中自检（每次回复前必查）

当你正在按本规则**为用户创建**某一编排 Skill 时，在写出本轮回复前，请自检：

- **若你尚未完成多轮提问、或尚未得到用户「可以出稿了」的确认**：本轮回复**必须**只包含「复述/概括 + 向用户提出的问题」或「询问是否要生成本次文档」，**不得**包含编排 Skill 的 SKILL.md 或 README 全文（或完整正文）。
- **若用户刚给出需求、你尚未提过任何问题**：本轮回复**必须**是「概括用户需求 + 第一批问题」，**禁止**直接输出任何编排 Skill 文档。
