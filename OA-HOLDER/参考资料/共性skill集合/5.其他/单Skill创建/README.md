# 我的 Skill 创建规则 — 使用说明

本 README 说明如何使用「我的 Skill 创建规则」这份 skill，以及本次创建时的决策记录与示例。

---

## 一、使用方式

### 何时会用到本 Skill

- 你想**按固定流程**创建新 skill：先和 AI 在对话里把想法全部聊清楚，再一次性得到「Skill 文档 + 配套 README」。
- 你希望**每次创建都先 ASK、再产出**，避免 AI 过早写稿或猜错重点。

### 推荐用法

1. **打开 ASK 模式（或仅讨论模式）**  
   在 Cursor 中进入只读/只讨论模式，避免 AI 直接改仓库。

2. **第一句话说明意图**  
   例如：  
   - 「我想按我的方式创建一个 skill：先聊清想法再出文档。」  
   - 「帮我用『先 ASK 再产出』的流程做一个 xxx 的 skill。」

3. **只描述「当前要做的这一个」skill**  
   只说这个 skill 要解决什么问题、给谁用、希望怎么用；不要在本轮里混入「参考别的 skill」或「和现有 skill 对比」。

4. **配合 AI 多轮提问**  
   AI 会不断追问目标、范围、方案取舍、边界条件，直到你确认「想法都摸清了」或「可以写了」。

5. **确认后再要交付物**  
   当你觉得需求已经讲完，说「可以出稿了」「就按这样写」等，AI 会一次性给出：  
   - 最终的 `SKILL.md`  
   - 配套的 `README.md`（含使用方式、本次确认/否定、案例与示例）。

### 和 AI 的配合要点

- **先聊想法，后要文档**：不要在第一句就要求「直接给我 SKILL.md」；先完成澄清再要产出。
- **确认与否定尽量说清楚**：你选了什么、否了什么，会被记进 README，方便以后回顾。
- **一次会话只做一个 skill**：若要做第二个，建议新开会话，再走一遍「先 ASK 再产出」。

## 二、本次创建中的确认与否定

以下来自设计「我的 Skill 创建规则」时的约定（仅限本 skill 的创建过程）。

### 确认采用的方案 / 观点

- **流程**：先用 ASK 模式说想法 → AI 分析并反复提问 → 摸清所有想法后再出最终内容。  
- **范围**：只针对「当前要创建的那一个 skill」，不考虑、不依赖已有 skill。  
- **产出物**：除 skill 文档本身外，必须有一份**配套 README**。  
- **README 内容**：必须包含「使用方式」「本次创建中的确认与否定」「使用案例」「结果示例」四类（或合并为「使用案例 + 结果示例」）。  
- **结束条件**：以用户确认「想法已摸清」或等价表述为信号，再产出 SKILL + README，不提前写稿。

### 否定的方案 / 不采用的写法

- **不**在未摸清想法前就写出完整 SKILL.md。  
- **不**在本流程中要求「参考或对比已有 skill」；本规则只服务于「当前这一个」的从零创建。  
- **不**在 README 里省略「确认与否定」或「使用案例 / 结果示例」；这些是本次约定的必备内容。

### 其他约定

- 「ASK 模式」指：只讨论、不直接改库；用户先表达想法，AI 只分析与提问。  
- 多轮提问可以包含：目标、范围、可选方案取舍、边界与约束；直到用户明确表示可以出稿为止。

## 三、使用案例

### 案例 1：做一个「需求一句话拆解」的 skill

- **用户说**：「我想按我的方式创建一个 skill：用来把客户的一句话需求拆解成结构化要点，先聊清我的想法再出文档。」  
- **预期过程**：AI 进入本规则流程，只讨论、不写稿；问清：拆解后的结构长什么样、谁用、输入输出、是否要示例等。用户多轮回答并最终说「可以写了」。  
- **预期结果**：得到一份「需求一句话拆解」的 SKILL.md，以及配套 README（含使用方式、本次确认/否定、案例与示例）。

### 案例 2：做一个「会议纪要转待办」的 skill

- **用户说**：「用先 ASK 再产出的方式，帮我做一个 skill：把会议纪要转成待办列表。」  
- **预期过程**：AI 提问：纪要格式（Markdown/纯文本）、待办要包含哪些字段、是否区分责任人/截止日期等。用户逐条确认后说「就这些，出稿吧」。  
- **预期结果**：得到该 skill 的 SKILL.md 与 README，其中 README 里有一节「本次创建中的确认与否定」，记录「要字段 A / 不要 B / 纪要格式以 Markdown 为准」等。

### 案例 3：只想讨论，暂不要文档

- **用户说**：「我想做一个 skill 但还没想好，先跟你聊聊。」  
- **预期过程**：AI 按本规则只做阶段一、二：分析 + 多轮提问，不产出 SKILL/README。  
- **预期结果**：会话结束时用户想法更清晰；若之后用户说「可以写了」，再在同一会话或新会话中产出文档（若新会话，可再次引用本 skill 并说「按我的方式出稿」）。

## 四、结果示例

### 产出物结构示例

按本规则完成一次创建后，你会得到类似结构：

```
.cursor/skills/<新 skill 名称>/
├── SKILL.md    # 含 name、description 与完整正文
└── README.md   # 含：使用方式、确认与否定、使用案例、结果示例
```

### README 节选示例（「确认与否定」）

```markdown
## 本次创建中的确认与否定

### 确认采用的方案
- 输出格式为 Markdown 列表，每条待办一行。
- 需要字段：事项、责任人、截止日期（可选）。

### 否定的方案
- 不自动解析自然语言日期（如「下周三」），只保留原文或留空。
- 不生成到 Jira/飞书，仅输出 Markdown。
```

### 何时算「摸清」的示例

- 用户：「就这些了，可以写 skill 和 README 了。」  
- 用户：「没问题了，按刚才说的出稿吧。」  
- 用户对最后一轮关键问题都做了选择，并说「没别的了」。

以上任一情况后，AI 应进入阶段三，产出 SKILL.md 与配套 README。
