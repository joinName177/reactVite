# 需求一致性审计专家 - 使用说明

本 Skill 旨在帮助产研团队在交付前进行“最后一道防线”的检查，**特别适用于研发团队已完成功能开发与测试，正式移交给交付团队时**。通过严格比对产品需求文档与测试用例（代表已实现的功能），确保最终交付给客户的功能与原始需求高度一致，无逻辑偏差。

## 使用方式

1.  **准备工作**：将你的产品需求文档（PRD）和测试用例/计划文档放在同一个文件夹中（可以有子文件夹）。
2.  **触发 Skill**：呼叫“需求一致性审计专家”。
3.  **提供输入**：直接发送该文件夹的**绝对路径**。
    > 例如：`/Users/username/project/docs/v1.2_feature/`
4.  **确认文件**：AI 会自动扫描并列出发现的文件，若分类准确，确认即可开始审计。
5.  **查看报告**：AI 将输出包含“详细对比表”的覆盖率报告。

## 本次创建中的确认与否定

在创建本 Skill 的过程中，用户明确了以下规则：

-   **输入形式**：✅ **文件夹路径**（不指定具体文件类型，由 AI 扫描）。
    -   *否定了*：直接粘贴文本、混合模式。
-   **核心目标**：✅ **查偏差**（研发开发的功能 vs 产品需求）。
-   **审计程度**：✅ **严格型**。不仅要有因果关系，还要求覆盖主流程 + 异常流程。
-   **偏差优先级**：
    1.  **逻辑冲突 (B)**：最严重，必须优先暴露。
    2.  **漏测风险 (A)**：其次。
    3.  **冗余/过度设计 (C)**：最后。
-   **输出形式**：✅ **表格对比法**。
    -   *否定了*：红黑榜法。

## 案例与示例

### 场景示例
研发团队已完成 v2.0 版本的开发并通过内部测试，现提交给交付团队进行验收。交付人员需要确认：研发实现的功能（通过其提供的测试用例体现）是否真的满足了产品最初的需求，是否存在“做歪了”或“漏做了”的情况，以确保给客户演示时不翻车。

**用户输入**：
> 请审计这个目录下的文档：`/docs/project/v2.0_payment/`

**AI 响应流程**：
1.  扫描目录，发现 `PRD_支付升级.md` 和 `TestPlan_支付v2.xlsx`。
2.  提取 PRD 中的 15 个功能点。
3.  提取 TestPlan 中的 40 个测试用例。
4.  逐一比对。

**结果示例（片段）**：

| 需求模块 | 需求简述 | 对应测试用例 | 审计结果 | 偏差/备注 |
| :--- | :--- | :--- | :--- | :--- |
| 支付路由 | 金额>1万走银联渠道 | Case-02: 大额支付走支付宝 | **【严重冲突】** | 需求指定走银联，用例预期走支付宝，逻辑不一致！ |
| 支付路由 | 支付失败重试 3 次 | (无) | **【缺失】** | 缺少针对重试机制的验证 |
