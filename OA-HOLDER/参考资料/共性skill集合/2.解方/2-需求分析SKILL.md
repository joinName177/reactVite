# 需求分析SKILL
 
 本技能用于需求原型设计输出，按六步法完成从背景到高保真原型的全链路交付。可在不同阶段独立调用，也可全流程执行。
 
 ## 何时使用
 - 需要从想法到可落地的原型与功能清单（含流程图/方案对比/确认/归档）时
 - 需要明确“支持…”能力表述与实现细节（字段/规则/交互/异常）以支撑评估与协作时
 - 需要在进入高保真前强制“确认”与质量校验时
 
 ## 不触发
 - 仅做轻量问答或粗略头脑风暴，未准备进入结构化分析与输出时
 - 用户明确要求使用“我的 Skill 创建规则”流程创建一个全新 Skill（该流程用于创建新 Skill，而非执行需求分析）
 
## 使用方式
- 输入来源：可基于“产品规划文档”，或直接以“我需要什么功能、谁使用、在什么场景”为起点；随后按用户功能分析拆解进入Step1的ASK收集与归纳
- 输入约束：明确平台/组件/接口可用性、优先级模型与关键场景
- 执行顺序：按 Step1→Step7；Step1需显式“已摸清/继续”后方可进入 Step2；Step3之后执行“必要性反证”并据结论决定是否进入Step4；可按需单独调用某一步（如仅需功能清单或方案对比）
- 决策节点：Step1、Step3.5与Step6均需显式“确认/继续”后方可进入下一步；若“不可行/不必要/延期”则停止或回Step5迭代
 
 ## 输入参数（建议）
- 业务背景：问题陈述、场景、目标指标、约束（法律/合规/资源/时间）
- 目标用户：分层、核心行为、关键动机、渠道与设备
- 竞品参考：目标竞品、差异点、借鉴点
- 技术与交付约束：平台（Web/iOS/Android/桌面）、可用组件/框架、数据接口可用性
- 项目目标与优先级：核心KPI、必须实现的P0项
- 品牌与风格（可选）：语气、视觉偏好、无障碍要求

## 执行流程

### Step1：头脑风暴
- 目标：在不定形阶段快速收集素材与想法，形成候选方向与边界初稿
- 交互规则（ASK模式）：
  - 模式：ASK，仅讨论不写稿；AI主动、连续多轮提问与归纳
  - 每轮包含：目标/范围/场景确认；可选方案取舍；边界与约束澄清
  - 暂停并等待用户指令：必须等待用户明确回复“已摸清/继续”后方可进入Step2
  - 门禁：必须得到用户明确反馈“已摸清/继续”后才进入 Step2；未明确则继续ASK与归纳，不进入后续步骤
  - 结束条件：当前会话内用户确认“已摸清”或等价表述
- 输出：头脑风暴要点清单与待确认项

输出模板：
```
【头脑风暴要点】
- 目标与价值：
- 范围与不做项：
- 场景与用户：
- 可选方案/取舍：
- 约束与风险：
- 待确认项：
```

确认模板（Step1门禁）：
```
【Step1确认】
- 是否已摸清：是/否
- 待补充要点：
- 下一步：继续Step2/继续头脑风暴
```

### Step2：核心背景与价值锚定
- 目标：明确业务背景、目标用户、核心价值，避免需求偏离核心方向
- 输出：背景-目标-价值 三维分析

输出模板：
```
【背景】
- 问题/机会：
- 场景与约束：
- 成功度量（KPI）：

【目标】
- 主要目标：
- 次要目标：

【价值】
- 对用户的核心价值：
- 对业务的核心价值：
- 不做事项（Out of scope）：
```

### Step3：用户故事与旅程
- 目标：将抽象用户转化为具体场景与旅程
- 输出：用户画像、全链路旅程、标准化用户故事

输出模板（画像）：
```
【用户画像】
- 核心属性（年龄/角色/设备/频次等）：
- 行为特征：
- 需求动机：
```

输出模板（旅程图）：
```
【全链路旅程】
- 阶段：发现 → 进入 → 使用 → 完成 → 留存
- 关键触点（渠道/页面/交互）：
- 场景痛点：
- 情绪波动：
- 期望动作：
```

输出模板（用户故事，Who-When-Do-Problem-Value）：
```
【用户故事】
- 谁：
- 在什么场景：
- 需要做什么：
- 解决什么问题：
- 获得什么价值：
```

### Step3.5：需求必要性反证
- 目标：在进入流程设计前，从价值/成本/风险等维度反证需求是否必要，避免无效投入
- 逻辑：
  - 收集反证点：已有替代方案、边际价值、成本与风险、合规约束、时间窗口、维护负担
  - 结论分级：必要 / 不必要 / 延期
  - 门禁：若结论为“不必要”，停止后续流程并归档；若“延期”，记录触发条件与复核时间后暂缓；若“必要”，进入Step4
- 输出模板：
```
【必要性反证】
- 场景与目标：
- 反证点：替代/边际价值/成本/风险/合规/时间窗/维护
- 结论：必要/不必要/延期
- 依据与数据：
- 后续动作：停止/暂缓复核时间/继续Step4
```

### Step4：推导流程与功能
- 目标：将用户旅程转化为系统流程逻辑，完成逻辑闭环并为后续方案确认与功能清单生成提供依据
- 输出：可视化流程图

流程图生成（Markdown Preview Mermaid Support）：
- 使用 Mermaid 语法绘制业务流程图（含正常流程、异常流程、边界场景）
- 标注核心节点、触发条件、反馈规则
- 利用 VS Code 插件 `Markdown Preview Mermaid Support` 实时预览流程图

示例（Mermaid）：
```
flowchart TD
  A[入口] -->|正常| B[核心节点]
  B --> C[完成]
  A -->|异常| E[异常处理]
  B -->|边界| F[边界场景]
```

 

### Step5：可行性设计方案输出（优化版）
- 目标：输出差异化方案，明确布局与交互思路，提供多维度选择，避免单一方案局限
- 输出：
  - 输出 2-3 版可行性设计方案，每版需明确：设计思路、方案内容（字段/按钮/核心元素描述）、布局描述（ASCII Art 页面框架）、方案亮点、适配场景
  - 制作方案对比表，从核心价值传达、业务匹配度、交互合理性、落地难度、开发成本 5 维度进行量化对比（含评分/权重）
  - 分析各方案优劣势，锚定 1 版建议方案并标注推荐理由；记录弃选原因
  - 推荐结论与理由必须置于“方案对比表”之后；同时给出不采纳方案的原因与缓解策略

方案模板（每版）：
```
【方案X】
- 设计思路：
  描述信息架构与交互主线（导航模式/关键路径/反馈策略）

- 方案内容（核心元素清单）：
  字段：
  - 名称 | 类型 | 必填 | 校验规则 | 默认值 | 数据来源/依赖
  按钮/交互：
  - 名称 | 动作 | 触发条件 | 反馈规则（成功/失败）| 异常处理
  组件：
  - 列表/卡片/表单/抽屉/弹窗等（含分页、排序、筛选、空态/加载态）

- 布局描述示例：（ASCII Art 页面框架）
+-----------------------------------------------------------------------+
| [Logo] 平台                        [当前模块: 销售商品管理]         |
+-----------------------------------------------------------------------+
| [筛选: 类别/状态]    🔍 搜索商品/规格...      [+新增]      |
+-----------------------------------------------------------------------+
| 序号|商品名称       | 规格名称        | 类别     | 状态   | 最近编辑人 | 操作            |
|----|--------------|-----------------|----------|--------|------------|--------------|
| 1  | 智慧食堂系统   | 年度授权(100端) | 软件   | 启用   | 张三    | [编辑][删除][渠道价] |
| 2 | 智能消费收银机 | 21.5寸/8G/128G  | 硬件   | 启用   | 李四    | [编辑][删除][渠道价] |
+-----------------------------------------------------------------------+
| [Footer] 共 128 条 | 每页 20 条 |      |
+-----------------------------------------------------------------------+

- 方案亮点：
  - 可用性/效率/一致性/可扩展性/易于实施等具体优势

- 适配场景：
  - 适合的业务场景、用户规模、设备/网络约束

- 风险与对策（可选）：
  - 风险点：依赖接口、性能峰值、学习成本
  - 对策：降级策略、缓存/预取、引导/空态优化
```

方案对比表（含评分/权重）：
```
评分说明：
- 1~5分：1=差 3=中 5=优
- 可配置权重（示例）：核心价值0.30 | 业务匹配度0.25 | 交互合理性0.20 | 落地难度0.15（反向）| 开发成本0.10（反向）
- 反向维度计算：反向分 = 6 - 原始评分（分数越低越好）

| 方案 | 核心价值 | 业务匹配度 | 交互合理性 | 落地难度(反向) | 开发成本(反向) | 权重加权总分 |
| ---- | -------- | ---------- | ---------- | -------------- | -------------- | ------------ |
| A    |    5     |     4      |     4      |       3        |       2        |   =Σ(分*权重) |
| B    |    4     |     5      |     3      |       4        |       3        |              |
| C    |    3     |     4      |     5      |       2        |       4        |              |
```

优劣势分析与决策模板：
```
【方案X优劣势】
- 优势：……
- 劣势：……
- 适配边界：……

【推荐方案】：A/B/C
【推荐理由】：
- 结合加权总分 + 关键路径可用性 + 风险可控性说明

【弃选方案原因】：
- 方案Y：具体不足/不匹配点 + 风险不可控项
```

### Step6：设计方案确认
- 目标：避免在未达成共识时进入高保真阶段
- 输出：显式确认与下一步指令
- 逻辑：
  - **暂停并等待用户指令**：必须等待用户明确回复“确认”或“继续”等指令后，方可进入 Step7
  - 若用户反馈“不可行”或提出修改建议，则收集具体优化点，循环回 Step5 直至达成共识
 
 确认模板：
 ```
 - 终版方案：A/B/C
 - 是否可行：是/否
 - 若否：优化建议要点（结构/交互/流程/文案/规则等）
 - 下一步：进入功能清单生成/优化后复核
 ```
 
结构化功能清单（分层表格，方案确认后生成）：
> **重要**：按“功能模块→功能名称→子功能/页面”分层拆解；“功能描述”必须以“支持...”开头且语言准确。字段/业务规则/交互/异常统一在“子功能详情”中描述，表格不展开细节。

```
| 功能模块 | 功能名称 | 子功能/页面名称 | 功能描述（支持...） | 优先级(P0/P1/P2) | 关联场景/用户故事 | 备注 |
| -------- | -------- | ---------------- | ------------------- | ---------------- | ------------------ | ---- |
|          |          |                  | 支持...             |                  | UJ-XX/场景标签     |      |
```

填表示例（参考）：
```
| 账户 | 注册 | 手机号注册页 | 支持用户通过手机号注册 | P0 | UJ-01（新用户注册） | 无 |
```

生成规则：
- 方案确认后，子功能详情需根据功能清单逐条拆分并生成，做到一一对应、无遗漏
- 子功能详情标题沿用清单中的“功能模块/功能名称/子功能（页面）名称”，可选增加“清单编号”用于映射

子功能详情模板（每个子功能/页面单独描述）：
```
【子功能详情】
- 功能模块：
- 功能名称：
- 子功能/页面名称：
- 子功能界面：关键组件/布局/状态（空态/加载/错误）
- 布局描述：ASCII框架/主区域/导航结构/响应式断点
- 逻辑规则：触发条件/校验规则/状态流转/异常处理
- 数据接口与依赖：入参/出参/权限/缓存/版本
- 可观测性：埋点事件/日志字段/告警规则
- 优先级：P0/P1/P2
- 关联场景/用户故事：编号
```

布局描述示例（ASCII Art 页面框架）：
```
+-------------------------------------------------------+
| Header: Logo | 导航 | 搜索                           |
+-------------------------------------------------------+
| Sidebar: 菜单/筛选             | Content: 列表/表单/详情 |
| - 分组1                        | - 空态/加载/错误状态    |
| - 分组2                        | - 分页/排序/筛选        |
+-------------------------------------------------------+
| Footer: 版权 | 帮助                                  |
+-------------------------------------------------------+
```

非功能需求（统一汇总）：
- 性能：QPS/延迟/并发/峰值与降级策略
- 兼容性：浏览器/操作系统/分辨率/网络条件
- 安全与合规：认证/授权/隐私/审计/数据保留
- 可用性与无障碍：键盘操作/ARIA/对比度/语言
- 可维护性：配置化/扩展性/版本与回滚策略


填表规范：
- 功能描述：一句话能力表述，必须以“支持...”开头
- 层级标注：每条记录需明确“子功能/页面名称”
- 优先级：按P0/P1/P2标注，P0覆盖关键路径闭环
- 关联场景/用户故事：指向Step3的用户旅程或用户故事编号
- 详细规则：字段/业务规则/交互/异常统一写在“子功能详情”，表格不展开
- 拆分要求：功能清单每条记录必须对应一个“子功能详情”，逐条生成，无遗漏

输出校验清单（质量检查）：
- 功能清单中的“支持...”描述清晰，无歧义
- 优先级与关联场景完整且一致
- 流程图包含正常/异常/边界分支并与功能清单相互映射
- 推荐结论与原因说明位于对比表之后，且理由完整
- 子功能详情已覆盖字段/业务规则/交互/异常处理与布局描述
- 非功能需求已明确性能/兼容性/安全等并与设计方案一致
- 必要性反证已完成并记录结论；若“不必要/延期”则已停止或暂缓并归档
- 子功能详情数量与功能清单的子功能（页面）条目数量一致，名称一一对应


### Step7：进入原型设计SKILL（高保真原型）
- 目标：在已确认的设计方案基础上生成可演示、可验收的单文件HTML高保真原型并预览
- 方式：调用“原型设计SKILL”，通过 CSS/HTML 拟态实现；本地预览端口 8084
- 输出：
  - 单文件HTML原型（含交互与样式设计tokens）
  - 规则标注：每组件悬浮可拖动按钮 + 右侧抽屉展示规则详情（data-rule-detail），支持就地编辑与批量注入（RULE_DICT）
 
- 后续归档（PRD可选）：
- 若需补充PRD文档，可沿用以下结构在原型完成后进行归档与版本管理（含版本号/责任人/审批记录/时间戳）

文档结构模板：
```markdown
# [项目名称] 产品设计文档

## 1. 核心背景与价值
- 背景：...
- 目标：...
- 价值：...

## 2. 用户与场景

## 3. 业务流程图（Mermaid）
- 流程图与核心节点说明：...

- 页面布局（ASCII）：...
- 核心元素与交互规则：
  - 字段：...
  - 按钮：...

## 5. 功能清单与子功能详情（确认后生成）
- 分层功能清单：模块/功能/子功能（页面）/支持…/优先级/关联场景
```

 ## 简要自检清单（产出前）
 - 用户已明确“确认/继续”或等价表示
 - Skill 的 name、description 与正文一致且无歧义
 - 功能清单符合“支持…”与“详细说明”两列标准
 - 流程图与功能清单一一映射，覆盖正常/异常/边界
 - 已完成“输出校验清单”的自检
