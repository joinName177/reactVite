# 客户一句话想法拆解 — 使用说明

本 README 说明本 Skill 的用法、本次按规范重新设计时的确认与否定、使用案例及结果示例。

---

## 一、使用方式

### 何时会用到本 Skill

- 客户只给了一句话想法，信息模糊，无法判断做给谁、做到什么程度、有哪些约束。
- 需要把客户原话转化为：目标用户、使用场景、要解决的问题、期望结果、已知约束等结构化信息，并区分原话/推断/待澄清。

### 推荐用法

1. **提供客户原话**：一句或几句客户描述；可选补充客户行业、公司规模。
2. **说明意图**：例如「帮我拆解一下这句话」「按目标用户、场景、问题、期望结果帮我结构化」。
3. **配合 AI 交互**：若信息不足，AI 会先列出待确认问题，待你补充或确认后再输出完整拆解。
4. **后续可接**：拆解后若有大量「待澄清」，可再让 AI 生成「客户追问清单」供下次沟通使用。

### 输入与输出

| 项目 | 说明 |
|------|------|
| **输入** | 客户原话（一句或几句）；可选：客户行业、公司规模。 |
| **输出** | 结构化拆解（目标用户、使用场景、要解决的问题、期望结果、已知约束）、隐含假设、待澄清项、已排除/已否定事项；各维度标注「原话/推断/待澄清」。 |

## 三、使用案例

### 案例 1：销售拿到客户一句需求

- **用户说**：「客户说想做一个能让销售自动生成报价单的系统，最好能和 CRM 打通。帮我拆解一下。」
- **预期过程**：AI 按本 Skill 执行拆解，输出目标用户（销售）、场景、问题、期望结果、约束，并标原话/推断，列出待澄清（如 CRM 类型、报价规则、触发方式）与隐含假设。
- **预期结果**：得到一份「想法拆解结果」Markdown，可直接用于内部评审或下次与客户确认。

### 案例 2：只有零散描述

- **用户说**：「我只有客户的一句想法：想做一个给门店用的订货 App，能对接我们现在的 ERP。先帮我拆成可操作的几块，并标出哪些是推断、哪些待澄清。」
- **预期过程**：AI 拆解为目标用户（门店）、场景（订货）、问题、期望结果（对接 ERP）、约束；推断与待澄清会明确标注；可能列出待澄清如 ERP 厂商、对接方式等。
- **预期结果**：结构化拆解 + 待澄清项列表，便于下次拜访前准备问题。

### 案例 3：信息不足时先澄清

- **用户说**：「客户说想做一个 XXX，帮我结构化一下。」（未提供具体描述。）
- **预期过程**：AI 发现信息过少，先向用户列出需确认的问题（如：客户原话原文、行业、规模等），待用户补充后再输出完整拆解。
- **预期结果**：先得到一组澄清问题；用户补充后得到完整想法拆解结果。

## 四、结果示例

### 产出物形态

执行本 Skill 后，将得到一份 Markdown 格式的「想法拆解结果」，包含：

1. **客户原话摘要**
2. **结构化拆解**（目标用户、使用场景、要解决的问题、期望结果、已知约束）
3. **隐含假设**
4. **已排除/已否定事项**

### 结构摘要示例（节选）

```markdown
### 🧩 想法拆解结果

#### 1. 客户原话摘要
"我们想做一个能让销售自动生成报价单的系统，最好能和 CRM 打通。"

#### 2. 结构化拆解
##### 目标用户
- 销售岗位（使用报价单的人员）。（原话）
##### 使用场景
- 销售在需要给客户出报价时使用。…
##### 要解决的问题
- 报价单制作耗时、易错。…
##### 期望结果
- 能自动生成报价单。与 CRM 数据打通。（原话）
##### 已知约束
- 需与现有 CRM 集成（品牌/类型未说明）。（原话）

#### 3. 隐含假设
* 客户已有或计划使用某款 CRM，且允许对接。…

#### 4. 已排除/已否定事项
* （按对话中实际排除项填写）
```

完整示例见本 Skill 文档末尾「示例」一节。
