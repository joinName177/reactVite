# 缺陷报告

编写规范、易复现的缺陷单。执行前可根据需要阅读项目 README.md 了解环境与术语。

## 原则

- 标题简洁且能概括问题本质；步骤可让开发无歧义复现。
- 包含预期结果与实际结果对比；环境与复现条件写清。
- 所有产出物保存到项目根目录下的 **测试** 文件夹。

## 输入

- 问题现象描述
- 复现步骤（若用户已提供则整理，否则根据描述补充）
- 环境信息：浏览器/系统/版本/账号等（如有）

## 输出

保存到 `测试/` 文件夹，建议文件名如 `缺陷-<模块>-<简短描述>.md` 或统一缺陷列表文件：

每条缺陷包含：**标题**、**严重程度**、**前置条件**、**复现步骤**、**预期结果**、**实际结果**、**环境信息**、**附件说明**（截图/日志/请求响应等）。

## 严重程度参考

| 级别 | 说明 |
|------|------|
| 致命 | 系统崩溃、数据丢失、核心功能不可用 |
| 严重 | 主要功能异常、无合理替代方案 |
| 一般 | 功能异常但有替代方案、或影响次要功能 |
| 轻微 | 界面、文案、易用性等问题 |

## 缺陷单模板

```markdown
# 缺陷：[简短标题]

- **严重程度**：致命/严重/一般/轻微
- **模块**：...
- **环境**：浏览器/系统/版本/账号等

## 前置条件
...

## 复现步骤
1. ...
2. ...
3. ...

## 预期结果
...

## 实际结果
...

## 附件
截图/日志/接口响应等说明
```

## 流程

1. **归纳标题**：用一句话说明「在什么情况下出现什么问题」。
2. **整理步骤**：按操作顺序编号，包含必要数据与操作路径。
3. **写清预期与实际**：便于开发判断是否修复正确。
4. **补充环境与附件**：便于复现与定位。

## 何时使用

用户提到「提 Bug」「写缺陷」「缺陷单」「报缺陷」或描述一个问题要求整理成规范缺陷时，自动应用本技能。
