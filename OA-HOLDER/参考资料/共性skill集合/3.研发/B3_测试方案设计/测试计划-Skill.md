# 测试计划

制定可执行的测试计划。执行前先阅读项目根目录的 README.md 了解项目目标与约束。

## 原则

- 计划基于已确认的需求与测试范围，不脱离实际。
- 采用简单、可控的方案（如先核心后外围、先功能后非功能）。
- 所有产出物保存到项目根目录下的 **测试** 文件夹。

## 输入

- 测试范围或测试需求分析结果
- 版本/迭代时间、可用人力、环境与工具约束（如有）

## 输出

保存到 `测试/` 文件夹，建议文件名如 `测试计划-<版本或迭代>.md`：

1. **测试目标**：本阶段要达成的质量目标。
2. **测试策略**：功能/回归/性能/安全等策略及优先级。
3. **测试范围**：在测/不在测模块与功能清单。
4. **资源与进度**：人力、环境、关键里程碑。
5. **风险与应对**：主要风险及缓解措施。

## 流程

1. **明确目标**：本版本/迭代的发布目标与验收标准。
2. **确定策略**：功能测试为主/回归为主、是否做性能或安全等。
3. **划定范围**：列出在测范围与明确排除项，避免范围蔓延。
4. **排期与资源**：按优先级分配任务与时间。
5. **风险识别**：依赖、环境、数据、进度等风险及应对。

## 输出模板

```markdown
# 测试计划 - [版本/迭代]

## 1. 测试目标
- 本阶段质量目标与验收标准

## 2. 测试策略
| 类型 | 策略说明 | 优先级 |
|------|----------|--------|
| 功能测试 | ... | 高/中/低 |
| 回归测试 | ... | ... |

## 3. 测试范围
- **在测**：模块/功能列表
- **不在测**：说明原因

## 4. 资源与进度
- 人力、环境、关键时间节点

## 5. 风险与应对
| 风险 | 影响 | 应对措施 |
|------|------|----------|
| ... | ... | ... |
```

## 何时使用

用户提到「测试计划」「测试方案」「测试策略」「迭代规划」或要求制定测试安排时，自动应用本技能。
