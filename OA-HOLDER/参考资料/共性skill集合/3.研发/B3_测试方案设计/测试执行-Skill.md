# 测试执行

指导如何执行测试、记录结果并规划回归。执行前先阅读项目 README.md 与当前测试计划/用例。

## 原则

- 按优先级执行（P0 → P1 → P2）；先主流程后分支。
- 结果记录清晰：通过/失败/阻塞/跳过及原因。
- 所有执行产出（结果记录、回归建议等）保存到项目根目录下的 **测试** 文件夹。

## 输入

- 测试用例文档（Markdown/CSV/Excel）
- 测试环境说明、版本信息（如有）
- 若为回归：上一轮结果或缺陷列表（如有）

## 输出

保存到 `测试/` 文件夹，建议文件名如 `测试执行结果-<版本>-<日期>.md`：

1. **执行摘要**：计划执行数、已执行、通过、失败、阻塞、跳过。
2. **结果明细**：用例 ID、标题、结果、备注（失败原因/阻塞原因）。
3. **回归建议**：失败用例对应的功能/模块、建议回归范围与优先级。

## 执行顺序建议

1. 执行所有 P0 用例（核心流程）。
2. 再执行 P1，随后 P2。
3. 失败用例：记录复现步骤与环境，便于编写缺陷单。
4. 阻塞用例：记录阻塞原因与依赖，单独跟踪。

## 结果记录模板

```markdown
# 测试执行结果 - [版本] - [日期]

## 执行摘要
| 项目 | 数量 |
|------|------|
| 计划执行 | ... |
| 已执行 | ... |
| 通过 | ... |
| 失败 | ... |
| 阻塞 | ... |
| 跳过 | ... |

## 结果明细
| 用例ID | 标题 | 结果 | 备注 |
|--------|------|------|------|
| TC-001 | ... | 通过/失败/阻塞/跳过 | ... |

## 回归建议
- 建议回归模块及原因
- 建议优先验证的用例
```

## 何时使用

用户提到「执行测试」「跑用例」「记录结果」「回归测试」「测试进度」或提供用例要求执行计划/结果记录时，自动应用本技能。
