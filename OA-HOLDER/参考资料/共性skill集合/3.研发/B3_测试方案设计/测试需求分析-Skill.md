# 测试需求分析

从需求/规格中提取可测试项，识别范围与缺口。**目标：不遗漏、不误读、功能点全部梳理**。执行前先阅读项目根目录的 README.md 了解项目背景。

**需求形式**：支持 **HTML 格式的需求文件**。需求若以 HTML 提供，可在页面中直接交互（如点击、展开、Tab 切换、筛选、链接跳转）；分析时需结合页面结构、可交互元素与正文内容一并提取功能点与测试点，不遗漏页面上的任何可操作项与展示项。

## 原则

- 严格基于已提供的需求文档，不臆造业务逻辑。
- 站在用户角度思考「如果我是用户，我需要什么」；以产品视角补全可能的需求缺漏并与用户确认。
- **充分考虑真实用户在实际使用中可能出现的情况**：误操作、重复点击/提交、中途离开再返回、弱网/断网、多标签页或后退/刷新、复制粘贴或超长输入、新用户摸索与老用户习惯差异等，在功能点与测试点中显式覆盖或标注，避免只按「理想路径」分析导致漏测。
- 所有产出物保存到项目根目录下的 **测试** 文件夹。
- **每次分析必须走完「功能点穷尽清单」「真实用户使用场景补充」与「理解一致性检查」**，避免遗漏与错误理解。

---

## 一、功能点穷尽清单（确保不遗漏）

按下列维度逐项遍历需求，凡涉及到的都要列入测试范围，并在输出中标注来源（如：来自 PRD 第 x 节 / 用户故事 US-xx）。

| 维度 | 检查项 | 说明 |
|------|--------|------|
| **角色/用户** | 每种角色、权限、用户类型 | 不同角色可见/可操作范围、菜单、按钮、数据隔离 |
| **页面/入口** | 每个页面、菜单、Tab、弹窗、抽屉 | 列表、详情、新增/编辑表单、批量操作入口 |
| **操作/动作** | 每个按钮、链接、提交、保存、删除、导入导出 | 含二次确认、成功/失败提示、跳转 |
| **接口/数据** | 每个接口、实体、关键字段 | 必填/可选、格式、长度、枚举、默认值、计算规则 |
| **状态/流程** | 每个状态、状态流转、审批节点 | 初始态、终态、回退、超时、并发提交 |
| **规则/约束** | 业务规则、校验规则、权限规则 | 谁在什么条件下能做什么、不能做什么 |
| **非功能** | 性能、兼容性、安全、易用性 | 若有明确指标或说明则纳入范围 |
| **关联/依赖** | 上下游模块、外部系统、配置项 | 依赖不可用时的表现、数据来源与去向 |
| **隐性需求/界面状态** | 空数据、加载中、错误提示、默认值、未授权/无权限 | 空列表、加载中态、错误态、默认值展示、无权限提示与跳转等，避免漏测 |

**自检**：上述每一行至少过一遍，文档中出现的都要在「测试范围清单」中有对应条目；若某维度在需求中未提及，在「需求疑义与遗漏」中记录「未说明 xxx，建议确认」。

## 一（补）、真实用户使用场景补充（确保贴近实际使用）

在穷尽清单之外，**从真实用户行为出发**补充易漏的测试点，并在「测试点列表」中单独或合并标注来源为「真实用户场景」。建议覆盖但不限于：

| 场景类型 | 考虑要点 | 示例 |
|----------|----------|------|
| **误操作/反悔** | 点错、取消、返回、关闭前未保存 | 误点删除后能否恢复、弹窗点取消/遮罩关闭、浏览器后退 |
| **重复/连续操作** | 连续点击、重复提交、双击提交按钮 | 防重复提交、按钮防抖、重复请求结果一致性 |
| **中断与恢复** | 中途离开、超时、会话过期、再回来继续 | 草稿保存、登录过期提示、未完成流程再进入 |
| **网络与环境** | 弱网、断网、请求超时、加载慢 | 加载中态、超时提示、部分失败时的提示与重试 |
| **输入与粘贴** | 复制粘贴、超长文本、空格/特殊字符、全角半角 | 粘贴带格式、长度截断、必填项仅空格、脚本/HTML 注入 |
| **多端与多态** | 多标签页、刷新、直接链接/书签进入、不同分辨率 | 多标签数据一致性、刷新后状态、深链参数、响应式布局 |
| **用户差异** | 新用户首次使用、老用户习惯、不同角色/权限 | 引导、默认值、无历史数据时的空状态、权限边界 |

若需求为 **HTML 页面**：结合页面上所有可点击、可输入、可展开的区域，逐一考虑上述场景在该元素上的表现，并纳入测试点。

## 二、理解一致性检查（确保不错误理解）

- **原文锚定**：每个测试点/功能点能对应到需求原文的一处或多处描述（可标注章节、段落或 AC 编号）。
- **术语统一**：在分析文档开头或附录列出「术语与缩写」，对易歧义词（如「审核/审批」「提交/保存」）给出本需求内的统一定义；有歧义时写入「需求疑义与遗漏」。
- **量化澄清**：对「尽快」「适量」「部分」等模糊表述，在疑义表中列出并建议改为可验证描述（如时间范围、数量范围）。
- **复述确认**：对复杂流程或规则，用「我的理解是：…」简要复述，若无法从原文唯一推断，则记为待确认。

---

## 三、输入

- **需求文档**：PRD、用户故事、验收标准（AC）；**支持 HTML 格式**——若提供 HTML 需求文件，可在页面中直接交互阅读，分析时需解析页面结构、可交互元素（链接、按钮、表单、Tab、折叠等）与正文内容，一并提取功能点并标注来源（如「来自 HTML 第 x 节 / id 或章节标题」）。
- 业务规则、字段约束、状态流转说明（如有）
- 原型图、接口文档、字典/枚举说明（若有请一并提供）

---

## 四、输出

保存到 `测试/` 文件夹，建议文件名如 `测试需求分析-<模块或版本>.md`：

1. **测试范围清单**：功能点列表、优先级（P0/P1/P2）、是否可测、**需求来源（章节/AC 编号）**。
2. **测试点列表**：每个功能对应的测试关注点（正常/异常/边界/数据/权限/状态/角色等），**与需求原文的对应关系**。
3. **需求疑义与遗漏**：模糊描述、缺失的验收标准、待与产品确认的问题列表；**穷尽清单中「未说明」项的汇总**。
4. **（可选）术语与缩写**：本需求内关键术语定义，避免后续用例与开发理解不一致。
5. **（推荐）需求追溯**：功能点 ↔ 需求章节/AC 编号的对应表或矩阵，便于后续用例覆盖复核与遗漏率控制。

---

## 五、流程

1. **解析需求**：提取关键动词、名词、业务实体与操作；若为 HTML，则结合页面结构（标题、段落、列表、表格）与可交互元素（链接、按钮、表单、Tab、折叠块等）逐项列出并标注来源（章节/AC 或 HTML 位置）。
2. **按穷尽清单遍历**：按「功能点穷尽清单」的维度逐项过一遍，列出所有功能点并标注来源。
3. **真实用户场景补充**：按「真实用户使用场景补充」表逐类过一遍，对当前需求涉及的功能补充误操作、重复操作、中断恢复、弱网、输入粘贴、多端多态、用户差异等测试点，并标注为「真实用户场景」。
4. **流程梳理**：理解主流程与分支/异常流程，标注关键节点与状态；检查是否有未描述的状态或分支。
5. **提取测试点**：为每个功能点列出「要验证什么」，并做理解一致性检查（原文锚定、术语、复述）。
6. **识别缺口**：标出描述不清、缺少 AC、可能遗漏的场景、穷尽清单中的「未说明」项及真实用户场景中的缺口，形成问题列表供评审。
7. **自检**：确认清单中每一维度已检查；确认真实用户场景已补充；确认无歧义未记录、无主观臆造逻辑。

## 七、何时使用

用户提到「需求分析」「测试范围」「测试点提取」「需求评审」「PRD 分析」或提供**需求文档（含 HTML 格式）**要求分析时，自动应用本技能。**执行时默认按「功能点穷尽清单」「真实用户使用场景补充」与「理解一致性检查」完成全部分析与自检；若需求为 HTML，需结合页面可交互元素与结构一并提取并标注来源。**
