# 测试用例设计

设计测试用例的覆盖策略与用例骨架，不直接写具体步骤。执行前先阅读项目根目录的 README.md 与相关需求文档。

## 原则

- 基于已确认的需求与测试点，不主观增加业务逻辑。
- 覆盖正向、异常、边界、状态流转与数据依赖。
- 生成测试计划时需严格考虑页面、功能、异常、兼容性、性能、安全、易用性等方面。
- **设计时充分考虑真实用户在实际使用中可能出现的情况**：误操作（点错、取消、返回）、重复点击/提交、中途离开再返回、弱网/断网、多标签或刷新、复制粘贴/超长输入、新用户与老用户不同路径等，为这些场景单独或合并设计用例骨架，避免仅按理想路径设计导致漏测。
- 所有产出物保存到项目根目录下的 **测试** 文件夹。
- **覆盖目标（支持将遗漏率控制在极低水平）**：每个测试点至少 1 条用例；P0 功能点至少覆盖「正向 + 异常」或「正向 + 边界」；多参数/多条件时考虑组合（如 pairwise）与负面用例（无效输入、无权限等）。

## 输入

- 需求文档或测试需求分析、测试点列表
- 业务规则、状态机、接口/字段约束（如有）

## 输出

保存到 `测试/` 文件夹，建议文件名如 `测试用例设计-<模块>.md`：

1. **设计策略**：等价类、边界值、场景法、状态迁移等使用说明；多参数时说明是否采用组合（如 pairwise）；是否显式包含负面用例（无效输入、无权限等）；**是否包含真实用户场景**（误操作、重复提交、中断恢复、弱网、多标签/刷新、异常输入等）。
2. **用例骨架**：用例编号、标题、类型（正向/异常/边界）、优先级、简要步骤与预期要点（非详细步骤）。
3. **覆盖矩阵**：需求/功能点与用例的对应关系，便于查漏；**每个测试点在矩阵中至少对应 1 条用例**。
4. **遗漏率自检**：输出「未覆盖测试点清单」；确认覆盖矩阵中无测试点遗漏；确认 P0 功能点已覆盖正向+异常或边界；**确认真实用户场景（若需求分析中已列出）均有对应用例**。

## 流程

1. **划分等价类**：输入/条件划分有效类与无效类。
2. **识别边界值**：取边界及边界外一点设计用例。
3. **梳理场景**：主流程、备选流程、异常流程、业务规则组合；多参数/多条件时考虑组合覆盖（如 pairwise）与负面用例；**从真实用户行为补充场景**（误操作、重复提交、中断后继续、弱网、多标签/刷新、异常输入、新老用户差异等）。
4. **生成骨架**：为每个设计点生成至少一条用例骨架（标题+类型+优先级+要点）；P0 功能点确保含正向+异常或边界；真实用户场景对应的测试点也需有对应用例。
5. **覆盖检查**：用矩阵核对需求/测试点是否都有对应用例；输出「未覆盖测试点清单」并补全用例直至无遗漏。
6. **遗漏率自检**：确认覆盖矩阵 100% 覆盖所有测试点；确认每个测试点至少 1 条用例；确认类型覆盖（正/异/边）满足上述覆盖目标。

## 输出模板

```markdown
# 测试用例设计 - [模块]

## 1. 设计策略
- 等价类：...
- 边界值：...
- 场景：主流程 / 异常 / 状态流转
- 真实用户场景（可选）：误操作、重复提交、中断恢复、弱网、多标签/刷新、异常输入等

## 2. 用例骨架
| 编号 | 标题 | 类型 | 优先级 | 步骤要点 | 预期要点 |
|------|------|------|--------|----------|----------|
| TC-001 | ... | 正向/异常/边界 | P0/P1/P2 | 1. ... 2. ... | ... |

## 3. 覆盖矩阵
| 需求/测试点 | 对应用例编号 |
|-------------|----------------|
| ... | TC-001, TC-002 |

## 4. 遗漏率自检
- [ ] 未覆盖测试点清单：（无则填「无」）
- [ ] 覆盖矩阵已 100% 覆盖所有测试点
- [ ] 每个测试点至少 1 条用例；P0 已覆盖正向+异常或边界
- [ ] 真实用户场景（若在需求分析中已列出）均有对应用例
```

## 何时使用

用户提到「用例设计」「等价类」「边界值」「场景设计」「测试覆盖」或提供需求要求设计用例时，自动应用本技能。
