### 🟡 B4-4 测试方案设计
> **归属板块**：产品/研发侧 | **主导角色**：测试负责人、测试工程师 | **工具**：Trae/Cursor + 测试方案设计 Skill 矩阵
> *⏳ 依赖 B1 需求契约化、B3 方案预设计/接口设计产出；可与 B4-2/B4-3 测试方案总纲衔接；输出供测试执行、缺陷管理与质量汇报使用*

*   **🎯 关键动作**：从需求与测试范围出发，完成测试需求分析、测试计划、用例设计、用例编写、执行指导、缺陷规范与测试报告全链路，形成可执行、可追溯的测试方案与产出。
*   **🤖 Skill**：按需调用下方 Skill 矩阵中的能力，完成从测试需求到测试报告的全流程设计。
*   **📥 输入**：需求文档 / PRD / 用户故事 / 测试范围、接口与原型说明（可选）
*   **📤 输出**：测试需求分析 + 测试计划 + 测试用例设计 + 测试用例文档 + 执行结果记录 + 缺陷报告规范 + 测试报告
*   **✅ 验证闭环**：**逻辑闭环**（测试范围无遗漏、用例覆盖可追溯、执行结果可统计、缺陷可复现、报告有据可依）

### 🛤️ Skill 组合路径建议（测试方案设计）

#### 🟢 场景一：新需求/新版本从 0 到 1（全链路）
*   **解决问题**：拿到新需求或新版本，从分析到报告完整走通。
*   **建议组合**：`测试需求分析` → `测试计划` → `测试用例设计` → `测试用例编写` → `测试执行` → `缺陷报告`（按需）→ `测试报告`

#### 🔵 场景二：仅做测试分析与计划
*   **解决问题**：先明确测什么、怎么测，暂不写具体用例。
*   **建议组合**：`测试需求分析` → `测试计划`

#### 🟣 场景三：已有设计，补写用例与执行
*   **解决问题**：已有测试点或用例设计，需要可执行用例与结果记录。
*   **建议组合**：`测试用例编写` → `测试执行` → `缺陷报告`（按需）→ `测试报告`

#### 🟠 场景四：执行后做总结与汇报
*   **解决问题**：执行完成，需要缺陷规范化与测试报告。
*   **建议组合**：`缺陷报告`（整理缺陷）→ `测试报告`
