# 测试用例编写

把用例设计转化为可执行的用例文档（步骤、预期、数据）。执行前先阅读项目 README.md 与测试用例设计或需求文档。

## 原则

- 步骤清晰、可执行；预期结果可判定通过/失败。
- 符合 SMART 原则（具体、可衡量、可达成、相关、有时限）。
- 生成测试用例时需严格考虑页面、功能、异常、兼容性、性能、安全、易用性等方面。
- 所有产出物保存到项目根目录下的 **测试** 文件夹。
- 支持输出 Markdown 表格或 CSV，便于导入 Jira、Excel 等。

## 输入

- 测试用例设计（用例骨架）或需求/测试点列表
- 期望格式：Markdown / CSV / Excel（如有）

## 输出

保存到 `测试/` 文件夹，建议文件名如 `<模块>-测试用例.md` 或 `<模块>-测试用例.csv`：

每条用例包含：ID、模块、用例标题、优先级、前置条件、测试步骤、预期结果、测试类型（功能/UI/接口等）、测试数据（如需要）、**需求来源/设计编号**（便于追溯与覆盖复核）。

## 格式规范

**Markdown 表格示例：**

| ID | 模块 | 标题 | 优先级 | 前置条件 | 测试步骤 | 预期结果 | 类型 | 需求来源/设计编号 |
|----|------|------|--------|----------|----------|----------|------|-------------------|
| TC-001 | 登录 | 手机号登录-成功 | P0 | 手机号已注册 | 1. 输入手机号 2. 输入验证码 3. 点击登录 | 登录成功，跳转首页 | 功能 | PRD 3.1 / TC-001 |

**步骤写法**：每步一行，编号 1、2、3…；包含操作与必要数据。  
**预期写法**：可验证的一句话或列表，避免模糊表述。

## 流程

1. **读取设计**：从用例设计或需求中取用例骨架与覆盖点。
2. **展开步骤**：为每条骨架编写详细步骤与预期，并填写需求来源/设计编号。
3. **补充数据**：需要特定数据时在步骤或单独列中写明。
4. **格式化**：按选定格式（Markdown/CSV）输出并保存到 `测试/`。
5. **编写后覆盖复核**：对照测试点列表或用例设计中的覆盖矩阵，检查是否每条测试点都有对应用例；若有遗漏则补写用例并再次复核，直至无遗漏。

## 何时使用

用户提到「编写测试用例」「导出用例」「生成用例」「用例转 Excel/Markdown」或提供设计/需求要求产出具体用例时，自动应用本技能。
