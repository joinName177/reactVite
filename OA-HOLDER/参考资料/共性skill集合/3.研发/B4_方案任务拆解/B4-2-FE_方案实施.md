### 🟠 B4-2-FE 方案实施（通用版）
> **归属板块**：研发侧（实现向） | **主导角色**：核心开发 / Tech Lead / 项目负责人 | **适用范围**：前端 / 后端 / 数据 / 测试 / 运维 / 脚本 & 工具开发 等所有「写代码」类工作  
> *⏳ 依赖 B3-1 方案预设计 & B3-2 接口设计 & B4-1 研发方案总纲产出；用于把「方案」落到可执行任务与可跟踪进度*

* **🎯 关键动作**：  
  无论是一个完整需求，还是当前手头的一坨改造/修复工作，  
  都通过固定套路完成三件事：
  1. **任务拆分**：把抽象需求 / 方案，拆成若干可执行的小任务（粒度 0.5–2 天为宜）  
  2. **清单编排**：在时间、人力约束下，排出「现在先做什么、后面分几批做什么」  
  3. **进度跟踪**：在每次写完代码后，基于任务完成度更新清单中的状态，下次一眼知道从哪接着做
* **🤖 Skill 组合**：按需调用下方 3 个通用 Skill —— 任务拆解、任务清单规划、任务进度管理。
* **📥 典型输入**：
  - 需求契约 / PRD / 用户故事 / Bug 描述
  - B3-1 方案预设计产物（前后端 / 数据 / 接口 / 安全等预设计）
  - B3-2 接口设计产物（API 契约、错误码规范、分页与过滤等）
  - B4-1 研发方案总纲（整体技术方案、里程碑、风险）
  - 当前已有代码仓库与历史实现说明（可选）
* **📤 典型输出**：
  - **任务拆分结构**：按模块 / 子系统 / 服务 / 页面 / 数据模型 / 技术基建等维度拆分的任务树
  - **可执行任务清单**：包含类型（FE/BE/DB/Test/Infra/Script 等）、优先级、预估工时、负责人/协作人、依赖关系、所属迭代/里程碑
  - **实施进度视图**：基于 `waiting_for_do` / `doing` / `done` 三态字段的任务状态视图，可映射到 Jira / Tapd / 飞书项目 / GitHub Projects 等工具
  - **风险与依赖清单**：需要其他人或其他系统配合的关键依赖点与时间窗
* **✅ 验证闭环**：
  - 每一项需求 / 方案点都能在任务清单中找到映射（**无遗漏**）
  - 任务粒度适中（建议单人 0.5–2 天可完成），不会出现“又大又笼统”的原子任务
  - 清单中的状态字段 `waiting_for_do` / `doing` / `done` 能在**每次写完代码后**被快速更新，真实反映当前进度与阻塞点

### 🛤️ Skill 组合使用路径

#### 🟢 场景一：新需求 / 新模块开发
* **解决问题**：从「有一份需求/方案」到「有一份拆分清单 + 排期 + 可追踪进度」。
* **组合推荐**：`Task Decomposer` → `Task List Planner` → `Execution Tracker`

#### 🔵 场景二：存量系统改造 / 技术债治理 / 性能优化
* **解决问题**：老系统上做改造或优化，改造点多且分散，难以控制范围与节奏。
* **组合推荐**：先用 `Task Decomposer` 快速拆出改造点与回归点 → 再用 `Task List Planner` 放入合适的迭代 / 里程碑中。

#### 🟣 场景三：紧急需求 / 线上生产问题修复
* **解决问题**：希望在不额外引入沉重流程的前提下，仍然对紧急修复保留最小可用的计划与进度记录。
* **组合推荐**：`Task Decomposer`（极简版，只拆出 0.5–1 天的小任务） → `Execution Tracker`（用三态字段快速跟进 + 事后补全记录）。
