## 一、项目概述

### 1.1 项目简介

OA-HOLDER 是一款基于 Electron 的桌面办公应用，采用 **Monorepo** 架构，整合工作台、任务管理、沟通协作、审批、专项管理、绩效、OKR 等企业 OA 核心功能。

### 1.2 架构特点

| 特点 | 说明 |
|------|------|
| **Monorepo** | 使用 pnpm workspace 管理多包，共享依赖与构建配置 |
| **分层架构** | 主进程 / 壳应用 / 共享包 / 业务模块 清晰分层 |
| **类型安全** | 端到端 TypeScript，IPC 通讯具备类型契约 |
| **模块化** | 共享包可复用，业务模块可独立构建与扩展 |

### 1.3 技术栈总览

| 层级 | 技术选型 | 版本 |
|------|----------|------|
| 运行时 | Electron | 39.x |
| 前端框架 | React | 19.x |
| 构建工具 | Vite | 6.x |
| 语言 | TypeScript | 5.9 |
| UI 组件库 | Ant Design | 6.x |
| 状态管理 | Zustand | 5.x |
| 路由 | React Router | 7.x |
| 图标 | Lucide React | 0.47x |
| 包管理 | pnpm | 9.14 |

## 四、核心模块说明

### 4.1 主进程（Electron）

| 模块 | 文件 | 说明 |
|------|------|------|
| 应用核心 | core/app-lifecycle.ts | 启动、退出、单例锁 |
| 窗口管理 | windows/window-manager.ts | 创建、销毁、聚焦窗口 |
| 窗口池 | windows/window-pool.ts | 预创建/复用窗口 |
| IPC 总线 | ipc/ipc-hub.ts | 注册与分发 IPC 处理器 |
| 共享状态 | ipc/shared-state.ts | 跨窗口状态同步 |
| 托盘 | native/tray.ts | 系统托盘菜单 |
| 自动更新 | native/auto-update.ts | 应用更新逻辑 |

### 4.2 壳应用（Shell）

| 模块 | 说明 |
|------|------|
| 侧边栏 | 主导航菜单，路由到各功能模块 |
| 顶栏 | 用户信息、通知、设置入口 |
| 路由 | React Router 7，按路径加载页面 |
| 状态 | Zustand：用户、UI、通知等全局状态 |

### 4.3 IPC 通讯

主进程与渲染进程通过 `@holder/ipc-bridge` 进行类型安全通讯：

- **主进程**：在 ipc-hub 中注册 handler
- **渲染进程**：通过 preload 暴露的 API 调用
- **类型契约**：在 `@holder/shared-types` 中定义 IPC 消息格式

    ├── 构建共享包（types → ipc-bridge, api, ui）

## 六、数据流与状态管理

### 6.1 状态分层

| 层级 | 方案 | 范围 | 示例 |
|------|------|------|------|
| 跨窗口共享 | IPC + SharedState | 主进程维护 | 用户信息、Token、通知计数 |
| 服务端数据 | API 请求 | 按需拉取 | 任务列表、审批列表 |
| 模块级 UI | Zustand | Shell 内 | 侧边栏折叠、主题、筛选条件 |
| 组件级 | useState | 组件内 | 输入值、弹窗开关 |

### 6.2 数据流向

```
用户操作 → Shell 组件
    │
    ├── 调用 @holder/api → 后端 API
    │
    └── 调用 @holder/ipc-bridge → 主进程
            │
            └── 主进程处理（窗口、托盘、数据库等）
```

### 7.1 独立模块（packages/modules/）

规划中的独立业务模块，可单独构建、按需加载：

- special-project（专项管理）
- performance（绩效）
- okr（OKR）
- plan（规划脑图）
- file-manage（档案管理）
- kpi（指标）
- normal-task（常规任务）

### 7.2 模块模板

`packages/modules/_template/` 提供模块脚手架，新模块可基于此创建。

### 7.3 创建新模块

1. 复制 `packages/modules/_template` 为 `packages/modules/your-module`
2. 修改 `package.json` 的 `name` 为 `@holder/your-module`
3. `pnpm-workspace.yaml` 已包含 `packages/modules/*`，无需额外配置
4. 执行 `pnpm install` 后即可开发

### 7.4 核心配置文件

| 文件 | 说明 |
|------|------|
| `pnpm-workspace.yaml` | 工作区包路径：apps/*、packages/shared/*、packages/modules/*、packages/tools/* |
| `turbo.json` | build 按依赖构建、dev 持久化、typecheck/test 依赖上游 |
| `tsconfig.base.json` | 共享 TypeScript 配置，各包 extends |
| `biome.json` | 代码风格与 Lint |

## 九、架构设计收益

本架构设计带来的主要收益如下：

### 9.1 开发效率

| 收益 | 说明 |
|------|------|
| **快速启动** | Vite 按需编译，开发启动 < 3s，HMR < 200ms |
| **并行开发** | Monorepo 下多人可同时开发不同应用/模块，依赖共享、冲突减少 |
| **增量构建** | Turborepo 支持增量构建，仅改动部分重新编译，全量构建时间可控 |

### 9.2 可维护性

| 收益 | 说明 |
|------|------|
| **职责清晰** | 主进程 / 壳应用 / 共享包 / 业务模块分层明确，单模块文件数可控 |
| **类型安全** | 端到端 TypeScript，IPC 契约统一在 shared-types，减少运行时错误 |
| **依赖透明** | 包依赖关系清晰，shared-types 无环，便于理解和重构 |

### 9.3 可扩展性

| 收益 | 说明 |
|------|------|
| **模块独立** | 业务模块可独立构建、按需加载，新模块不影响现有功能 |
| **共享复用** | UI、API、IPC 等抽成共享包，新应用/模块直接复用，避免重复开发 |
| **技术升级** | 共享包升级后，依赖方统一受益，升级成本集中、可控 |

### 9.4 运行与安全

| 收益 | 说明 |
|------|------|
| **进程隔离** | 主进程与渲染进程分离，独立模块可单独窗口，崩溃隔离 |
| **上下文隔离** | preload 暴露最小 API，降低 XSS 等安全风险 |
| **现代运行时** | Electron 39 + Node 22，安全补丁及时、性能更好 |

### 9.5 团队协作

| 收益 | 说明 |
|------|------|
| **统一规范** | 共享包约束技术栈与代码风格，新人上手更快 |
| **独立交付** | 模块可独立测试、发布，支持按业务线拆分迭代 |
| **文档集中** | 架构文档与代码结构一致，便于沟通与交接 |
