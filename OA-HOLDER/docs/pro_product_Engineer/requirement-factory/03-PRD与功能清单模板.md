## 二、功能清单模板

| 模块 | 功能点 | 子功能/页面 | 描述（支持…） | 优先级 | 原型关联 | 验收标准 |
|------|--------|-------------|---------------|--------|----------|----------|
| 模块 A | 功能 1 | 子功能/页面名 | 支持用户通过 xxx 完成 xxx | P0 | 原型页-01 | 可测试标准 |
| 模块 A | 功能 2 | 子功能/页面名 | 支持 xxx | P1 | 原型页-02 | 可测试标准 |
| 模块 B | 功能 1 | 子功能/页面名 | 支持 xxx | P0 | 原型页-03 | 可测试标准 |

**描述规范**：功能描述必须以「支持…」开头；字段/业务规则/交互/异常在子功能详情中描述。

### 优先级说明

- **P0**：必须有，MVP 核心
- **P1**：应该有，首版重要
- **P2**：可以有，后续迭代

## 三、5W2H 需求澄清表

| 维度 | 问题 | 回答 | 来源 |
|------|------|------|------|
| Who | 目标用户是谁？ | | 原话/推断/待澄清 |
| What | 要解决什么问题？ | | 原话/推断/待澄清 |
| Why | 为什么要做？ | | 原话/推断/待澄清 |
| Where | 在什么场景下使用？ | | 原话/推断/待澄清 |
| When | 使用时机、频率？ | | 原话/推断/待澄清 |
| How | 用户期望如何操作？ | | 原话/推断/待澄清 |
| How much | 优先级、投入、预期效果？ | | 原话/推断/待澄清 |

**来源说明**：原话 = 需求方原文；推断 = 基于常识/行业推断；待澄清 = 需进一步确认。**已排除事项**：在需求澄清纪要中单独记录需求方明确排除的方案或「不做」的事项。
