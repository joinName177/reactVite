# 从一句话需求到原型与功能清单的完整流程

> 产品经理将一句话需求转化为可交付原型及功能清单的标准流程指南  
> **说明**：需求输出阶段暂不考虑运维、发布相关内容  
> **规则**：每个步骤均含自检，遇到任何问题需停留确认，解决后再进行下一步

---

## 一、需求澄清与理解（需求分析）

### 1.1 5W2H 拆解

| 维度 | 关键问题 |
|------|----------|
| **Who** | 目标用户是谁？有哪些角色？ |
| **What** | 要解决什么问题？核心价值是什么？ |
| **Why** | 为什么要做？业务/用户动机？ |
| **Where** | 在什么场景下使用？ |
| **When** | 使用时机、频率？ |
| **How** | 用户期望如何操作？ |
| **How much** | 优先级、投入、预期效果？ |

**【自检】** 5W2H 拆解完成
- [ ] 7 个维度均有明确回答
- [ ] 回答与原始需求一致、无矛盾
- [ ] **遇到问题**：暂停，与需求方确认后再进行下一步

### 1.2 需求澄清会

- 与需求方（业务/客户）确认理解是否一致
- 明确边界：做/不做、先做/后做
- 记录假设和待确认点

**【自检】** 需求澄清会完成
- [ ] 需求方已确认理解一致
- [ ] 边界已明确（做/不做、先做/后做）
- [ ] 假设和待确认点已记录
- [ ] **遇到问题**：暂停，与需求方确认后再进行下一步

**【自检】** 用户旅程完成

- [ ] **遇到问题**：暂停，补充或与需求方确认后再进行下一步

## 三、功能规划

### 3.1 功能脑暴

- 围绕目标列出所有可能功能
- 不急于砍需求，先穷举

**【自检】** 功能脑暴完成
- [ ] 功能列表已穷举，无遗漏
- [ ] 每个功能与目标可对应
- [ ] **遇到问题**：暂停，补充或与需求方确认后再进行下一步

### 3.2 MVP 筛选

- 用「必须有 / 应该有 / 可以有」分类
- 结合资源、时间确定 MVP 范围

**【自检】** MVP 筛选完成
- [ ] 功能已按三类分类
- [ ] MVP 范围已确定且与资源、时间匹配
- [ ] **遇到问题**：暂停，与需求方确认后再进行下一步

### 3.3 功能优先级

- 使用 MoSCoW、RICE、Kano 等模型排序
- 明确 P0 / P1 / P2

**【自检】** 功能优先级完成
- [ ] P0 / P1 / P2 已明确
- [ ] 优先级与 MVP 范围一致
- [ ] **遇到问题**：暂停，与需求方确认后再进行下一步

## 五、原型设计

### 5.1 低保真原型（线框图）

- 页面布局、模块划分
- 主要元素位置、信息层级
- 工具：纸笔、Figma、Axure、墨刀等

**【自检】** 低保真原型完成
- [ ] 所有关键页面已有线框图
- [ ] 布局、模块、信息层级清晰
- [ ] 与信息架构一致
- [ ] **遇到问题**：暂停，修正或与需求方确认后再进行下一步

### 5.2 交互说明

- 点击、跳转、弹窗、反馈
- 加载、空态、错误态
- 表单校验、提交反馈

**【自检】** 交互说明完成
- [ ] 主流程交互已说明
- [ ] 加载、空态、错误态已设计
- [ ] 表单校验规则已定义
- [ ] **遇到问题**：暂停，补充或与需求方确认后再进行下一步

### 5.3 高保真原型（可选）

- 视觉风格、配色、组件
- 更接近最终界面，便于评审和开发理解

**【自检】** 高保真原型完成（如执行）
- [ ] 视觉风格、配色已确定
- [ ] 与低保真逻辑一致
- [ ] **遇到问题**：暂停，与设计/需求方确认后再进行下一步

### 5.4 原型评审

- 与业务、设计、开发一起过稿
- 收集反馈并迭代

**【自检】** 原型评审完成
- [ ] 业务、设计、开发已参与评审
- [ ] 反馈已收集并完成迭代
- [ ] 无遗留争议问题
- [ ] **遇到问题**：暂停，解决争议后再进行下一步

## 流程总览（简化版）

```
一句话需求
    ↓
需求澄清（5W2H、澄清会）
    ↓
用户研究（画像、旅程、竞品）
    ↓
功能规划（脑暴、MVP、优先级）
    ↓
信息架构 + 流程图
    ↓
原型设计（低保真 → 高保真 → 评审）
    ↓
功能清单 + PRD
    ↓
交付开发
```

---

## 实用建议

1. **先小范围验证**：复杂需求可先做简单原型或 Demo 做验证
2. **持续对齐**：每个阶段都跟业务、设计、开发同步，避免后期大改
3. **版本化**：用 v1.0、v1.1 管理 PRD 和原型，方便追溯
4. **复用模板**：建立 PRD、功能清单、原型规范模板，提高效率
