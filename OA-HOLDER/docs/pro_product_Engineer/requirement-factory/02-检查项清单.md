# 产品需求检查项清单

> 从一句话需求到原型与功能清单过程中的易遗漏检查项，用于查漏补缺  
> **说明**：需求输出阶段暂不考虑运维、发布相关内容  
> **自检规则**：使用本清单自检时，如遇任何问题需停留确认，解决后再进行下一步

---

## 一、合规与风控

| 检查项 | 说明 |
|--------|------|
| 合规性 | 是否涉及 GDPR、个人信息保护法、行业法规等 |
| 隐私与数据安全 | 敏感数据收集、存储、脱敏、加密要求 |
| 权限与审计 | 权限模型、操作审计、数据访问控制 |
| 法律风险 | 文案、功能是否存在法律或合规风险 |

---

## 二、技术可行性

| 检查项 | 说明 |
|--------|------|
| 技术可行性 | 与研发确认实现难度、技术选型、技术债 |
| 系统依赖 | 依赖的现有系统、接口、第三方服务 |
| 数据模型 | 核心实体、字段、关系是否清晰 |
| 接口设计 | 前后端接口约定、数据格式、错误码 |

---

## 三、边界与异常

| 检查项 | 说明 |
|--------|------|
| 空状态 | 无数据、无权限时的展示与引导 |
| 错误态 | 网络异常、接口失败、超时等的提示与处理 |
| 加载态 | 骨架屏、loading、进度反馈 |
| 边界值 | 最大/最小长度、数量限制、并发限制 |
| 异常流程 | 中断、取消、超时、重试等分支流程 |

---

## 四、业务与数据

| 检查项 | 说明 |
|--------|------|
| 成功指标 | 如何衡量功能是否成功（转化率、使用率等） |
| 埋点与统计 | 关键行为埋点、漏斗、报表需求 |
| 数据迁移 | 老数据是否需要迁移、兼容策略 |
| 默认与初始化 | 默认配置、初始数据、示例内容 |

---

## 五、体验与可访问性

| 检查项 | 说明 |
|--------|------|
| 无障碍 | 是否需满足 WCAG 等无障碍标准 |
| 多端适配 | PC / 移动 / 平板、分辨率、横竖屏 |
| 多语言 | 是否支持 i18n、语言切换 |
| 引导与帮助 | 新手引导、帮助文档、FAQ |

---

## 六、协作与交付

| 检查项 | 说明 |
|--------|------|
| 验收标准 | 每个功能点的可测试、可验收标准 |
| 变更影响 | 对现有功能、用户、流程的影响 |
| 培训与沟通 | 内部培训、用户通知 |
| 文档交付 | PRD、接口文档、测试用例 |

---

## 七、需求质量自检

| 检查项 | 说明 |
|--------|------|
| 需求完整性 | 是否覆盖主流程、异常流程、边界情况 |
| 需求一致性 | 与业务目标、产品定位是否一致 |
| 需求可测试性 | 是否可写出清晰的测试用例 |
| 需求可追溯 | 功能与原始需求、业务目标可对应 |

---

## 八、评审前快速检查清单

```
□ 5W2H 是否都回答清楚？
□ 用户画像和场景是否明确？
□ 功能优先级是否达成共识？
□ 原型是否覆盖主流程和异常流程？
□ 空态、错误态、加载态是否设计？
□ 合规、隐私、权限是否考虑？
□ 技术可行性是否评估？
□ 成功指标和埋点是否定义？
□ 验收标准是否可执行？
```

---

## 九、按项目类型补充的检查项

### ToB / 企业级

- 多租户、权限、审批流、审计、合规、集成

### ToC / 高并发

- 性能、容量

### 涉及支付

- 支付流程、对账、退款、风控

### 涉及内容

- 审核、敏感词、举报、版权

### 涉及社交

- 举报、拉黑、隐私、消息推送

---

## 使用建议

可将以上检查项整理进 PRD 模板或评审 checklist，在需求澄清、原型评审、开发前评审等阶段按需勾选，减少遗漏。**勾选时如遇问题，需停留确认后再继续。**
