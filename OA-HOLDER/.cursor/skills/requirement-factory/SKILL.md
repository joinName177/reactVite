# 需求工厂 (Requirement Factory)

> 一句话需求 → 原型与功能清单的完整方法论  
> 整合 pm-flow：P0 澄清 → 19 步流程 → 可选自动化验收  
> 需求输出阶段暂不考虑运维、发布相关内容

## 触发条件

当用户消息满足以下**任一**情况时，按本技能执行：

- 消息以 **`一句话需求:`** 或 **`一句话需求：`** 开头
- 消息包含 **`按默认协议`**、**`按 PM 流程`**、**`按 product-manager 流程`**
- 用户明确说 **「帮我分析需求」**、**「走需求流程」**、**「出需求卡和原型」**
- 用户通过 **`/pm-flow`** 显式调用

## 核心规则

1. **严格顺序**：按步骤编号执行，不可跳步
2. **自检必过**：每步自检项全部通过后方可进入下一步
3. **问题停留**：自检不通过或遇疑问时，**暂停**，与需求方确认后再继续
4. **输出前置**：下一步依赖上一步的输出物
5. **门禁确认**：S08 后需完成「需求必要性反证」；S06、S15、S19 需用户明确「确认/继续」后方可进入下一步
6. **P0 先行**：第一轮回复**只输出 P0 澄清问题清单**，不得直接给方案或代码；等用户回答后再进入 S01

## 张小龙视角原则（贯穿全流程）

- 用户只做一件事：把问题说清楚
- 三步内完成，默认值优先
- 用完即走，不做负担
- 少即是多，先删再加
- 先验证再扩张

## P0 澄清问题清单（第一轮必出）

在进入 S01 前，**第一轮回复只输出 P0 澄清问题清单**，不得直接给方案或代码。清单涵盖：

| 类别 | 关键问题 |
|------|----------|
| 用户与场景 | 目标用户是谁？主/次角色？使用场景与频率？ |
| 对象与范围 | 核心对象是什么？做/不做的边界？先做/后做的范围？ |
| 主路径 | 用户完成目标的主流程是什么？关键步骤？ |
| 规则与边界 | 业务规则、约束、合规要求？ |
| 验收口径 | 如何判断「做对了」？成功标准？可量化指标？ |

**禁止行为**：用户发了「一句话需求: xxx」时，不得直接给实现方案或写代码，必须先问 P0 澄清问题。

## 流程概览

```
一句话需求
    ↓
P0 澄清问题清单 → 用户回答
    ↓
需求澄清（5W2H、澄清会；原话/推断/待澄清；已排除事项）
    ↓
用户研究（画像、旅程、竞品）
    ↓
功能规划（脑暴、MVP、优先级）
    ↓
需求必要性反证（门禁：必要→继续 / 不必要→停止 / 延期→暂缓）
    ↓
信息架构 + 流程图 + 状态流转
    ↓
方案对比与推荐（2–3 版方案、推荐结论）
    ↓
原型设计（低保真 → 交互说明 → 高保真可选 → 逻辑漏洞检测 → 评审）
    ↓
功能清单 + 非功能需求 + PRD
    ↓
交付物确认 → 交付开发
```

## 5W2H 拆解（S01）

| 维度 | 关键问题 |
|------|----------|
| Who | 目标用户是谁？有哪些角色？ |
| What | 要解决什么问题？核心价值是什么？ |
| Why | 为什么要做？业务/用户动机？ |
| Where | 在什么场景下使用？ |
| When | 使用时机、频率？ |
| How | 用户期望如何操作？ |
| How much | 优先级、投入、预期效果？ |

**自检**：7 个维度均有明确回答；回答与原始需求一致、无矛盾。

**来源标注**：每个维度的回答需标注来源——「原话」（需求方原文体现）、「推断」（基于常识/行业推断）、「待澄清」（需进一步确认）。若存在「待澄清」项，须先与需求方确认后再进入 S02。

**已排除事项**：若需求方在澄清中明确排除某方案、否定某选项或约定「不做」某类需求，须在 S02 需求澄清纪要中单独记录，便于后续不重复讨论。

## 需求必要性反证（S08 后门禁）

在进入 S09 信息架构前，必须完成需求必要性反证，避免无效投入。

**反证维度**：已有替代方案、边际价值、成本与风险、合规约束、时间窗口、维护负担。

**结论分级**：
- **必要**：进入 S09
- **不必要**：停止流程，归档结论与依据
- **延期**：记录触发条件与复核时间，暂缓后待复核

**输出模板**：
```
【需求必要性反证】
- 场景与目标：
- 反证点：替代/边际价值/成本/风险/合规/时间窗/维护
- 结论：必要/不必要/延期
- 依据与数据：
- 后续动作：继续 S09 / 停止并归档 / 暂缓（复核时间：xxx）
```

**门禁**：仅当结论为「必要」时方可进入 S09；结论为「不必要」或「延期」时，**暂停**并等待需求方决策。

## 方案对比与推荐（S11.5）

在 S11 状态流转完成后、S12 低保真原型前，输出 2–3 版可行性方案并给出推荐结论。

**每版方案结构**：设计思路、方案内容（功能总览/布局要点）、方案亮点、适配场景。

**方案对比表**：从核心价值、业务匹配度、交互合理性、落地难度、开发成本等维度评分（1–5 分），计算加权总分。

**推荐结论**：必须置于对比表之后，明确推荐方案及理由；对不采纳方案说明弃选原因与缓解策略。

**门禁**：需用户明确「确认可行/继续」选定方案后，方可进入 S12 低保真原型。

## 功能清单结构

| 模块 | 功能点 | 子功能/页面 | 描述（支持…） | 优先级 | 原型关联 | 验收标准 |
|------|--------|-------------|---------------|--------|----------|----------|
| 模块 A | 功能 1 | 子功能/页面名 | 支持… | P0 | 原型页-01 | 可测试标准 |

**描述规范**：功能描述必须以「支持…」开头，表述能力与边界，无歧义。字段/业务规则/交互/异常统一在子功能详情中描述。

**优先级**：P0 必须有（MVP 核心）、P1 应该有（首版重要）、P2 可以有（后续迭代）。

**拆解原则**：每个功能点可独立开发、可测试；描述清楚输入、输出、规则、异常；功能清单每条记录必须对应一个子功能详情，逐条生成，无遗漏。

### 功能清单生成与自检（八步法，S16 必须执行）

产出功能清单时，必须按以下八步执行，并**产出自检表**（作为独立小节写入交付物）：

| 步骤 | 动作 |
|------|------|
| 第一遍 | **方案文档**：逐节扫 02-需求方案/方案对比，转化为功能条；每节至少对应 0～N 条，不可整节跳过 |
| 第二遍 | **流程图**：对每个节点（含分支、异常）写出对应功能点；每个节点 → 至少一条功能或注明「不产出」 |
| 第三遍 | **隐性功能**：逐项确认并写入：加载态、错误处理、空状态、权限校验、撤销/回滚（可逆时） |
| 第四遍 | **原型对照**：若有关联原型，对照页面元素与交互，遗漏补入并标注来源 |
| 第五遍 | **CRUD 与主对象**：对主对象按增删改查过一遍；不做的在「本期不做」中显式注明 |
| 第六遍 | **关联需求**：梳理数据来源/去向、共用能力、权限与范围 |
| 第七遍 | **必检项**：无禁词（管理/系统/能力/处理/支持）、原子化、可追溯、优先级与字数符合 |
| 第八遍 | **专家审查**：调用 product-expert 做功能清单专项审查；根据专家输出修正并更新自检表 |

**功能清单禁词表**：❌ 拒绝「管理」「系统」「能力」「处理」「支持」；✅ 接受「创建订单」「查询列表」「修改状态」等动词+宾语。

**自检表模板**（必须写入交付物）：

| 检查项 | 说明 | ✓ |
|--------|------|---|
| 方案文档覆盖 | 每节已扫，无整节遗漏 |  |
| 流程图节点覆盖 | 每个节点已映射功能或注明不产出 |  |
| 隐性功能覆盖 | 加载/错误/空状态/权限/撤销 已逐项确认 |  |
| 原型对照（如有） | 原型可见交互在清单中有对应条 |  |
| CRUD 与主对象 | 增/删/改/查已过一遍；不做的已显式注明 |  |
| 关联需求梳理 | 已含数据来源、去向、共用能力、权限 |  |
| 必检项 | 无禁词、原子化、可追溯、优先级符合 |  |

## 反思能力（S11.5 方案选定后必须执行）

在给出推荐方案后，必须做一次**反思 pass**：

- **质疑多余概念**：方案里是否引入不必要的「模式」「开关」或步骤？
- **以「选择」为中心**：多对象操作优先「先选对象，再选动作」，避免多一步
- **统一单人/多人路径**：单人/多人尽量用同一套逻辑
- **成功标准再检视**：用「最清晰、最简便」重新数步骤；若可省一步，在最终方案里体现

**产出**：若有调整，用 1～3 句话说明「反思结论 + 与原方案的差异」；若无需调整，可写「经反思，原方案已满足最简路径，无需变更」。

## 产品专家评审（product-expert）

在以下节点**建议**调用 product-expert 技能进行专家评审：

- **S08 后、S09 前**：反证结论与方案空间的一致性
- **S15 前**：原型与逻辑漏洞检测前的方案完整性
- **S16 功能清单定稿前**：**必须**执行功能清单专项审查；根据专家输出修正 03，更新自检表

专家评审输出原则：只写问题不写通过项；每条 ≤30 字；总输出 ≤400 字。

## 评审前快速检查清单

执行 S15 原型评审或 S19 交付物确认前，必须逐项勾选：

```
□ 5W2H 是否都回答清楚？
□ 用户画像和场景是否明确？
□ 功能优先级是否达成共识？
□ 原型是否覆盖主流程和异常流程？
□ 空态、错误态、加载态是否设计？
□ 合规、隐私、权限是否考虑？
□ 技术可行性是否评估？
□ 成功指标和埋点是否定义？
□ 验收标准是否可执行？
□ 逻辑漏洞检测是否完成（流程断点、权限、状态机、异常分支）？
```

## S15 逻辑漏洞检测要点

在原型评审前，建议进行逻辑漏洞检测，覆盖：

- **导航与返回**：页面进出一致性、返回后状态保持
- **权限与角色**：越权操作、角色切换后数据隔离
- **状态机**：不可达/无终止/循环；跨状态操作合法性
- **异常与空态**：请求失败/无数据/部分成功的页态与引导
- **并发与重复提交**：多次点击/网络抖动下的幂等与提示
- **表单校验**：前后端一致、跨字段校验、错误定位

---

## 易遗漏检查项（9 类）

1. **合规与风控**：GDPR、隐私、权限、审计、法律风险
2. **技术可行性**：实现难度、系统依赖、数据模型、接口设计
3. **边界与异常**：空态、错误态、加载态、边界值、异常流程
4. **业务与数据**：成功指标、埋点、数据迁移、默认与初始化
5. **体验与可访问性**：无障碍、多端适配、多语言、引导与帮助
6. **协作与交付**：验收标准、变更影响、培训、文档交付
7. **需求质量**：完整性、一致性、可测试性、可追溯性
8. **ToB 补充**：多租户、审批流、审计、集成
9. **ToC/支付/内容/社交**：按项目类型补充

**原型生成规范**（`prototype.html` 或项目适配格式）：

- 所有需求完成后，在 `05-原型与交互/` 下生成高保真原型
- **设计角色**：以**顶级设计师**身份进行设计，追求生产级、有辨识度的界面，避免通用 AI 美学
- **技术栈开放**：根据设计需求选择最合适的实现方式——可选用 React、Vue、Tailwind、设计系统等，以达成最佳视觉与交互效果
- **设计原则**：明确美学方向（极简/极繁/复古未来/有机/奢华/ playful 等）；字体、配色、动效、布局需有明确设计意图；避免 Inter、Roboto、紫白渐变等俗套组合
- **交付形式**：单文件 HTML 可直接预览；若使用框架，提供可运行的项目结构及启动说明

**设计层原则（顶级设计师角色）**：在 S12–S14 原型设计阶段，以顶级设计师身份执行：

- **美学方向**：明确选择极简/极繁/复古未来/有机/奢华/playful/ editorial 等方向，并贯穿始终
- **差异化**：避免 Inter、Roboto、Arial、紫白渐变等俗套；字体、配色、动效需有独特辨识度
- **空间与层次**：布局可不对称、重叠、留白；背景可用渐变、纹理、噪点增强氛围
- **动效**：适度使用入场动画、悬停反馈、滚动触发，提升体验而非堆砌

**规则**：执行 S19 时，先创建目标文件夹，再将各步骤产出写入对应文件；若用户未指定需求关键词，可从 S01 的 What/Who 提炼。

若项目使用 `原型/模块/` 结构，可产出以下命名：

- `01-需求卡-<模块名>.md`：背景、用户、成功标准、约束
- `02-需求方案-<模块名>.md`：多方案对比、流程图、推荐方案
- `03-功能清单与验收标准-<模块名>.md`：原子化功能列表、验收用例 TC、自检表
- `原型/模块/<模块>/验收/acceptance.spec.mjs`：Playwright 验收脚本
- `原型/模块/<模块>/验收/验收问题表-<需求名>.md`：失败用例与复现步骤

**原型与 03 的强制关联**：原型实现范围**仅**以 03 功能清单为准，逐条实现或标注「本期不做/占位」；交付时必须在 03 中产出「原型实现情况对照」表。

## 自动化验收（S19 后可选）

原型生成后，若用户确认「执行自动化验收」，按以下步骤执行：

| 步骤 | 动作 |
|------|------|
| Step 11 | 在 `验收/` 下生成 Playwright 验收脚本（`acceptance.spec.mjs`），TC 与 03 验收标准一一对应；生成 `验收/README.md` |
| Step 12 | 用户确认后自动运行验收；收集通过/失败列表，失败项保留错误信息、截图或录屏路径 |
| Step 13 | 根据失败结果生成**验收问题表**（`验收/验收问题表-<需求名>.md`），关联 TC ID、功能清单项、预期/实际/严重程度/复现步骤；将问题表交给研发修复 |

**前置检查**：环境可用性、入口可用性、用例映射性、证据路径、执行边界已说明并获用户确认。

**失败分支**：自动化执行失败时，必须输出失败 TC、错误信息、复现步骤、证据路径、影响功能清单项；无法直接路由给研发时，仍交付完整问题表。
