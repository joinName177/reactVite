# 产品专家评审技能

> 供 requirement-factory 在关键节点调用，用于方案与功能清单的专项审查

## 输出原则（必须遵守）

- **只写问题，不写通过的项**。通过的检查项一律不输出。
- **每条 ≤30 字**，直指问题与修正建议。
- **总输出 ≤400 字**。问题多时只列最关键的 5～7 条。
- 无问题则输出：「评审通过，无关键问题。」

---

## 评审维度与关键问题

按以下维度检查，**仅输出有问题的项**：

### 流程合规
- 反证未在方案设计前完成 → 指出
- 反证结论为「不必要/可延后」却进入后续方案设计 → 指出
- 未做反思 pass → 指出

### 产出完整性
- 缺价值主张/业务目标/关键指标
- 缺成功标准或不可验收
- 用户对象未区分主/次，或「所有人都是目标用户」
- 缺反证结论，或未考虑错误后路（撤销/删除）
- 未先列约束，或未考虑 IA 深度（过载、空状态）
- 缺方案对比表，或某方案缺流程图
- 缺假设与待澄清、缺「本期不做」

### 逻辑一致性
- 方案无法支撑成功标准
- 主角色/主场景在方案中无对应
- 流程图主路径 >7 步、节点含糊

### 风险
- 技术/业务依赖未明确，缺降级策略
- 假设与待澄清模糊

### 功能清单专项审查（输入 02 + 03 时执行）

当输入包含 02-需求方案 与 03-功能清单与验收标准 时，按以下维度检查：

| 检查项 | 说明 |
|--------|------|
| 02→03 映射 | 02 各节（含流程图节点）是否已映射到 03 功能条或显式注明「不产出」 |
| CRUD 与主对象 | 主对象的增删改查是否齐全，不做的是否在「本期不做」中显式注明 |
| 可交互操作与 TC | 可交互操作是否均在验收标准中有对应 TC |
| 隐性功能 | 加载态、错误处理、空状态、权限校验、撤销/回滚（如适用）是否已列出 |
| 关联需求 | 「关联与依赖」小节是否完整（数据来源/去向、共用能力、权限） |
| 本期不做一致性 | 02 与 03 的「本期不做」是否一致 |
| 03 结构完整性 | 关联与依赖、功能清单、本期不做、验收标准、自检表是否齐全 |
| TC 与功能清单映射 | P0 功能是否有对应 TC；是否存在孤儿 TC 或孤儿功能 |

**输出格式**（功能清单专项审查）：
```markdown
## 功能清单专项审查结论

- **评审通过**：无关键遗漏。
- **或** 存在问题（仅列有问题的项，每条 ≤30 字）：
  1. [02 §X.X] 未映射到功能清单 → 建议补「xxx」
  2. 主对象「xxx」缺「删」→ 建议补或注明本期不做
  3. 隐性功能「空状态」未单独列出 → 建议补
  ...
```
