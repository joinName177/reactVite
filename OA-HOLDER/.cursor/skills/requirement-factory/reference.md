## 二、功能清单模板

| 模块 | 功能点 | 子功能/页面 | 描述（支持…） | 优先级 | 原型关联 | 验收标准 |
|------|--------|-------------|---------------|--------|----------|----------|
| 模块 A | 功能 1 | 子功能/页面名 | 支持用户通过 xxx 完成 xxx | P0 | 原型页-01 | 可测试标准 |
| 模块 A | 功能 2 | 子功能/页面名 | 支持 xxx | P1 | 原型页-02 | 可测试标准 |
| 模块 B | 功能 1 | 子功能/页面名 | 支持 xxx | P0 | 原型页-03 | 可测试标准 |

**描述规范**：功能描述必须以「支持…」开头，表述能力与边界；字段/业务规则/交互/异常在子功能详情中描述。

**优先级说明**：P0 必须有（MVP 核心）、P1 应该有（首版重要）、P2 可以有（后续迭代）。

## 三、5W2H 需求澄清表

| 维度 | 问题 | 回答 | 来源 |
|------|------|------|------|
| Who | 目标用户是谁？ | | 原话/推断/待澄清 |
| What | 要解决什么问题？ | | 原话/推断/待澄清 |
| Why | 为什么要做？ | | 原话/推断/待澄清 |
| Where | 在什么场景下使用？ | | 原话/推断/待澄清 |
| When | 使用时机、频率？ | | 原话/推断/待澄清 |
| How | 用户期望如何操作？ | | 原话/推断/待澄清 |
| How much | 优先级、投入、预期效果？ | | 原话/推断/待澄清 |

**来源说明**：原话 = 需求方原文体现；推断 = 基于常识/行业推断；待澄清 = 需进一步确认。存在「待澄清」时须先确认再进入 S02。

**已排除/已否定事项**（记录在需求澄清纪要中）：需求方明确排除的方案、否定的选项或约定「不做」的事项。

## 七、原型设计规范（S12–S14 参考）

- **低保真**：线框图、布局、信息层级清晰；与信息架构一致
- **高保真**：视觉风格、配色、组件状态（hover/focus/active/disabled）明确；追求生产级、有辨识度的界面
- **设计角色**：以**顶级设计师**身份设计，明确美学方向，避免通用 AI 美学（如 Inter、Roboto、紫白渐变）
- **技术栈开放**：根据设计需求选择最合适的实现——原生 HTML/CSS、React、Vue、Tailwind、设计系统等均可，以达成最佳效果
- **交互覆盖**：页面切换、弹窗显隐、表单校验、空态/加载态/错误态
- **规则标注**（可选）：关键控件可增加 `data-rule-detail` 或 Tooltip 说明校验/默认值/逻辑规则，便于评审与验收
- **风格参考**：可遵循 Ant Design 等设计系统；色板、间距、圆角、阴影统一；亦可采用独特字体、配色与动效

## 八、验收衔接说明

交付物确认通过后，若需进入产品验收阶段，可衔接「产品验收 SKILL」：

- 输入：HTML 原型、规则标注、功能清单
- 输出：验收清单（CSV/Markdown）、Gherkin 测试用例骨架、缺陷分级
- 用户侧验收：按模块执行「前置条件→操作步骤→期望结果」，在验收结果列选择通过/不通过

## 九、各步骤自检项速查

| 步骤 | 自检项 |
|------|--------|
| S01 | 7 个维度均有明确回答；回答与原始需求一致、无矛盾；来源（原话/推断/待澄清）已标注 |
| S02 | 需求方已确认理解一致；边界已明确；假设和待确认点已记录；已排除事项已单独列出 |
| S03 | 至少 1 个典型用户画像；特征、目标、痛点、使用场景均有描述 |
| S04 | 主流程关键步骤已列出；每步触点、情绪、痛点已标注 |
| S05 | 至少 1 个竞品或现状已分析；可借鉴点、可差异化点已明确 |
| S06 | 功能列表已穷举；每个功能与目标可对应 |
| S07 | 功能已按三类分类；MVP 范围与资源、时间匹配 |
| S08 | P0/P1/P2 已明确；与 MVP 范围一致 |
| 门禁 | 需求必要性反证已完成；结论为「必要」；若「不必要/延期」已归档或暂缓 |
| S09 | 功能模块划分清晰；页面层级已定义；导航结构已设计 |
| S10 | 主流程已绘制；异常流程、分支已覆盖；泳道图（如有）已明确 |
| S11 | 关键对象已识别；各状态及流转规则已定义 |
| S11.5 | 2–3 版方案已输出；对比表与推荐结论已明确；用户已确认选定方案 |
| S12 | 所有关键页面有线框图；布局、模块、信息层级清晰；与 IA 一致 |
| S13 | 主流程交互已说明；加载、空态、错误态已设计；表单校验已定义 |
| S14 | 视觉风格、配色已确定；与低保真逻辑一致 |
| S15 | 业务、设计、开发已参与；逻辑漏洞检测已完成；反馈已迭代；无遗留争议 |
| S16 | 模块、功能点、子功能、描述（支持…）、优先级、原型关联均已填写；与规划、原型对应；可独立开发、可测试 |
| S17 | 性能、安全、兼容性、可扩展性已定义（按需） |
| S18 | PRD 各章节均已写入；与前述输出一致 |
| S19 | 7 项交付物均已产出；可交付开发 |

## 十一、pm-flow 风格文档结构（可选）

当使用 `01-需求卡`、`02-需求方案`、`03-功能清单` 命名时：

### 01-需求卡
- 背景、用户、成功标准、约束

### 02-需求方案
- 用户故事、核心指标
- 业务流程图（Mermaid）
- 功能详情（按页面/子功能：界面描述、逻辑规则、数据埋点）
- 非功能需求
- 验收标准（TC 表格）

### 03-功能清单与验收标准
- 功能清单表格（完成 | 序号 | 模块/功能描述 | 详细说明 | 关联成功标准 | 优先级）
- 本期不做
- 关联与依赖（数据来源、去向、共用能力、权限）
- 验收标准 TC 表
- 功能清单自检表（八步法检查项）
